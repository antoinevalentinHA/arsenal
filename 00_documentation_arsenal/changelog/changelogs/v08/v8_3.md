## 📌 Chauffage — Souveraineté consignes & neutralisation ViCare

- Adoption explicite du modèle **HA maître absolu / ViCare esclave passif** pour les consignes.
- Abandon du paradigme “sync bidirectionnelle” :
  - suppression des synchronisations inverses **ViCare → HA** (consignes),
  - maintien d’une orchestration **monodirectionnelle HA → ViCare**.
- Introduction de **scripts d’application souverains** (points d’écriture matériels uniques) :
  - `script.chauffage_appliquer_consigne_confort`
  - `script.chauffage_appliquer_consigne_reduite`
- Refonte des automatisations de sync :
  - l’automation devient **déclencheur / orchestration pure**,
  - l’exécution réelle est **déléguée aux scripts** (anti-duplication d’autorité).
- Défense active par observabilité :
  - création de capteurs métier d’**incohérence consignes** (cloud vs souveraineté HA),
  - ajout d’une automation de **réalignement consignes** déclenchée uniquement sur incohérence qualifiée + stabilité.

### Signal net
- ViCare est rétrogradé au rang de **canal non fiable** : il peut dériver, mais ne peut plus influencer la vérité métier.
- Chaîne durcie : **diagnostic → orchestration → exécution (unique)**.

### Risques
- Toute référence historique aux anciennes automatisations de sync (chemins/IDs) est **obsolète**.
- Robustesse dépendante de la qualité des lectures “locales” (`sensor.*_local`) utilisées pour qualifier les divergences.

## 🔐 Alarme / Wi-Fi — Apprentissage BSSID piloté par signal métier

- Refonte du mécanisme d’apprentissage BSSID :
  - passage d’une logique “calcul dans l’automation” à un modèle canon :
    **capteur métier (diagnostic) → automation (action) → helper (mémoire)**.
- Simplification de l’automation d’apprentissage (`10120000000016`) :
  - déclenchement principal sur `binary_sensor.wifi_nouveau_bssid = on`,
  - lecture des candidats via attributs,
  - écriture uniquement en cas d’ajout réel,
  - journalisation uniquement si apprentissage effectif.
- Réorganisation des templates système BSSID :
  - structuration en sous-dossier `12_template_sensors/system/bssid/`,
  - ajout d’un capteur “nouveau BSSID” dédié.

### Signal net
- L’alarme ne “déduit” plus : elle reçoit un **signal métier qualifié**, vérifiable, auditable (candidats exposés).

### Risques
- Toute dépendance UI/diagnostic à l’ancien template `telephones_bssid.yaml` doit pointer vers le nouvel emplacement.
