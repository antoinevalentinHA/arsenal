## 1. Rupture majeure — Migration source `programme_chauffage`

**`12_template_sensors/chauffage/programme.yaml`**

Refonte complète de la logique de dérivation.

- **Avant :** dérivation depuis `input_select.chauffage_dernier_mode_decide` (mémoire décisionnelle Arsenal)
- **Après :** dérivation depuis `sensor.boiler_heating_setpoint` (lecture réelle via boiler bridge)

Le capteur n'est plus une **projection de décision** mais une **interprétation du réel**.

Comparaison par tolérance flottante pour éviter les faux écarts IEEE :

```jinja2
{% set eps = 0.05 %}
{% if (setpoint - reduced) | abs <= eps %}  {# → Eco #}
{% elif (setpoint - comfort) | abs <= eps %} {# → Confort #}
{% else %}                                   {# → Inconnu #}
```

Attributs de diagnostic mis à jour :

| Attribut supprimé | Attribut remplaçant |
|---|---|
| `mode_arsenal` | `boiler_heating_setpoint` |
| `application_en_cours` | `consigne_reduite_locale` |
| `raison` | `consigne_confort_locale` |
| — | `mode_derive_depuis` |

> ⚠️ L'état `Inconnu` sera plus fréquent qu'avant — c'est intentionnel.
> Il signale une désynchronisation, une consigne externe ou une dérive système.
> Ce n'est pas un bug, c'est un signal.

## 2. Consigne appliquée — vérité exposée chaudière

**`12_template_sensors/chauffage/consigne_appliquee.yaml`**

Simplification majeure.

- **Avant :** reconstruction locale depuis `sensor.programme_chauffage` + sélection conditionnelle entre `input_number.chauffage_consigne_reduite` et `input_number.chauffage_consigne_confort`
- **Après :** lecture directe de `sensor.boiler_heating_setpoint`, sans projection ni reconstruction

```
Avant : programme Arsenal → sélection locale → consigne exposée
Après : boiler bridge → lecture directe → consigne exposée
```

> ⚠️ Arsenal ne reconstruit plus la consigne.
> La chaudière devient la source de vérité exposée.

Autres corrections :
- Disponibilité conditionnée à la disponibilité du bridge (plus de dépendance au programme local)
- Ajout du `unique_id` manquant (`temperature_consigne_appliquee_locale`)

## 4. UI — Template `boiler_decision_ack`

**`button_card_templates/40_dashboards/boiler/20_decision/boiler_decision_ack.yaml`**

Un ACK n'est pas une décision — c'est un statut transactionnel. Le socle est corrigé en conséquence.

- Socle : `socle_decision_72` → `socle_status_72`
- Ajout d'un bloc `icon` avec mapping sémantique complet par état ACK
- Ajout d'un bloc `state_display` avec libellés français normalisés

| État | Icône | Label |
|---|---|---|
| `applied` | `mdi:check-circle-outline` | Applique |
| `accepted` | `mdi:clock-outline` | Accepte |
| `rejected` | `mdi:close-circle-outline` | Rejete |
| `timeout` | `mdi:timer-alert-outline` | Timeout |
| `unknown / unavailable` | `mdi:help-circle-outline` | Inconnu |
| autre | `mdi:alert-circle-outline` | Anormal |

> ⚠️ `accepted` n'est pas un succès final. Seul `applied` représente un succès appliqué.

> 📌 Les états ACK représentent des étapes d'un cycle transactionnel : émission → acceptation → application → retour.

Suppression du bloc `🧩 USAGE` (exemple retiré du header).
