## Vue globale

v11.1.7 est une version **structurellement active** — pas un patch mineur.
Trois axes simultanés :

1. **Externalisation totale des décisions** (volets, présence, garage)
2. **Normalisation des flux météo → action** (pipeline pluie/volets décomposé)
3. **Ajout d'une couche "qualité du monde réel"** (Netatmo, Bluetooth)

Le système bascule dans une phase **data-driven** : les sensors portent la vérité métier, les
scripts sont des exécutants purs, les automations assurent l'orchestration. Corollaire : la
qualité des sensors devient critique.

## [PRÉSENCE] Refonte structurelle de `high_accuracy_on` (10210000000007)

**Changement de paradigme de déclenchement :** abandon des `binary_sensor.approche_securite_*`
individuels au profit du trigger natif `platform: zone` sur `zone.approche_securite`.

```yaml
platform: zone
entity_id: person.valentin  # ou person.constance
zone: zone.approche_securite
event: enter
```

- Trigger migré : `binary_sensor → platform: zone (event: enter)` — déclenchement structurel,
  plus robuste que la médiation par capteurs intermédiaires, aligné sur le modèle natif
  Home Assistant
- Renommage du trigger id `antoine → valentin` (alignement `person.valentin`)
- Suppression de la condition anti-rebond `trigger.from_state != 'on'` (rendue caduque par le
  trigger zone)
- Variable `cible_notify` : réécriture en `>-` (sans pollution whitespace terminale) — correction
  du bug latent de non-activation : la pollution whitespace rendait la condition `!= 'none'`
  faillible
- En-tête enrichi : ajout section `⚙️ GARANTIES` — sans dépendance aux libellés UI
  (`friendly_name`), mapping explicite `trigger.id → cible`, sans pollution whitespace terminale

## [PRÉSENCE] Normalisation `binary_sensor.presence_famille_unifiee`

Refonte de l'en-tête de `12_template_sensors/presence/global.yaml` — le capteur était décrit
comme un simple agrégat avec temporisation ; il est désormais contractualisé :

- Rôle explicité : capteur canonique de consommation, agrège
  `binary_sensor.presence_famille` et `binary_sensor.presence_famille_securite`
- Temporisation formalisée : `delay_on` absorbe les activations fugitives, `delay_off` absorbe

  capteur et de ses sources

## [VOLETS/PLUIE] Refonte du pipeline — séparation stricte décision/exécution

Architecture cible atteinte sur ce domaine :

```
décision      → template sensor
orchestration → automation
exécution     → script (pur)
```

### `script.volets_fermeture_execute` — simplification radicale

Le script abandonne toute logique décisionnelle interne. Il devient un **exécuteur pur paramétré** :

- Suppression de toutes les variables de contexte internes (présence, pluie, autorisation, cibles
  séjour/chambres)
- Suppression des branches A/B (présence ON/OFF) et de la notification globale
- Ajout du champ `covers` (required, domain: cover) — la liste est entièrement fournie par
  l'appelant
- Logique réduite à un `repeat.for_each` avec deux garde-fous : état actionnable + idempotence
  (`!= closed`)
- Référence normative : `CONTRAT_PLUIE_VOLETS.md v2.2.1 — section 5.6`

### `sensor.cibles_volets_pluie_sejour` — suppression de la contrainte fenêtres

- Suppression de la dépendance aux `binary_sensor.contact_sejour_*`
- Invariant séjour désormais : `intention_pluie_forte + autorisation` uniquement (protection
  globale inconditionnelle — le séjour se ferme sans condition sur l'état des fenêtres)
- Suppression de l'attribut `exposition_fenetres_entity_ids` et du `default_entity_id`
- Politique : `sejour_protection_totale`
- Référence : `CONTRAT_PLUIE_VOLETS.md v2.2.1 — sections 4.2, 5.3`

### `sensor.cibles_volets_pluie_chambres` — ajout de la politique présence/verrou

- Politique présence désormais portée par le sensor (et non plus par le script exécutif) :
  conditions `absence` (`presence_famille_securite == off`) et `verrou`
  (`input_boolean.fermeture_volets_pluie`) en garde-fous du sensor
- Attributs de debug ajoutés : `presence`, `verrou`
- Politique : `chambres_conditionnelles`
- Référence : `CONTRAT_PLUIE_VOLETS.md v2.2.1 — sections 4.1, 5.3`

**Note d'asymétrie assumée :** séjour = protection totale (sans condition fenêtre) ;
chambres = conditionnelles (présence + verrou). Cette divergence est contractuelle et
intentionnelle. L'attribut `politique` dans chaque sensor expose ce comportement pour le debug.

### `11_automations/meteo/pluie/pluie_volets_sejour.yaml`

- Suppression des conditions `presence_securisee` et `autorisation_sejour` de l'automation
  (déléguées au sensor)
- Condition `cibles non vides` requalifiée en optimisation
- `states(...) == 'on'` → `is_state(...)` sur les garde-fous pluie forte
- Référence contrat ajoutée en en-tête : `CONTRAT_PLUIE_VOLETS.md v2.2.1 — sections 4.2, 5.7`

## [GARAGE] Refonte de `script.garage_toggle`

Remplacement de la logique générique à paramètre `action` (on/off/toggle) par une **bascule pure
basée sur l'état logique mémorisé** :

- Suppression du champ `fields.action` et de toutes les variables de résolution d'action effective
- Deux branches `choose` explicites :
  - `OFF→ON` : `switch.turn_on` + `delay 200ms` + `input_boolean.turn_on`
  - `ON→OFF` : symétrique
- La commande physique passe de `button.press` (SwitchBot) à `switch.turn_on/off` sur
  `switch.lumiere_garage`
- Source de vérité : `input_boolean.garage_light_state` (inchangée)
- Parti pris assumé et contractualisé : pas de vérité physique indépendante → vérité logique
  mémorisée fait foi
- En-tête réécrit : principe de bascule, contrainte matérielle (relais Zigbee sans capteur
  indépendant)

**Dette technique identifiée :** en cas de dérive physique/logique (bouton physique, lag Zigbee),
le toggle s'inverse. Un `script.garage_resync` (reset forcé de `input_boolean.garage_light_state`
sur l'état réel) est à prévoir en v11.1.x suivante.

## Points de vigilance

**Sensors volets — défaillance silencieuse possible.** Si `sensor.cibles_volets_pluie_*` produit
une liste vide pour une mauvaise raison (sensor indisponible, évaluation partielle), aucun volet
ne se fermera sans erreur visible côté automation. Les attributs de debug (`autorisation`,
`presence`, `verrou`, `politique`) permettent le diagnostic. À terme : envisager un attribut
`coherent` booléen dans chaque sensor, consommable comme garde-fou explicite dans les automations
appelantes.

**Recorder — nouveaux sensors diagnostic.** Les `*_score` Netatmo (wifi, RF) sont classés
Population B : les exclure du recorder pour éviter spam logbook et croissance DB. Les `*_degrade`
et `*_ko` sont Population A et peuvent être enregistrés normalement. Vérifier l'allowlist avant
déploiement.

**Garage — dérive physique/logique.** La simplification est cohérente avec le contrat (vérité
logique mémorisée), mais un appui physique non tracé ou un lag Zigbee peut désynchroniser
`input_boolean.garage_light_state` de la réalité. Un `script.garage_resync` est identifié comme
dette technique à traiter en v11.1.x suivante.

**Pipeline volets — asymétrie séjour/chambres.** Comportement intentionnel et contractuel, exposé
via l'attribut `politique` de chaque sensor. À documenter explicitement dans
`CONTRAT_PLUIE_VOLETS.md` si ce n'est pas encore le cas.
