# ==========================================================
# 🌆 ÉCLAIRAGE JARDIN — ARCHITECTURE À DEADLINES
# ==========================================================

## 🧠 Nouveau modèle causal

Suppression progressive du modèle :

- "binary_sensor temps réel comme autorité"

Remplacement par :

- faits persistants explicites
- deadlines reconstructibles
- causalité événementielle durable

L’extinction du jardin ne dépend plus :
- de `last_changed`
- d’un front temporel fugitif
- d’une réévaluation implicite

Elle dépend désormais uniquement de deadlines persistantes.

---

## ➕ Nouveaux helpers persistants

Ajout :

- `input_datetime.jardin_soir_allumage_effectif`
- `input_datetime.jardin_soir_extinction_deadline`
- `input_datetime.sejour_inactivite_deadline`
- `input_datetime.jardin_sejour_derniere_activite`

Ces helpers deviennent les faits de référence du domaine.

---

## 🧠 Nouvelle doctrine Arsenal

Le contrat Jardin évolue officiellement :

Avant :
- décisions métier stateless

Après :
- décisions métier dérivables d’états persistants explicites

Cette évolution formalise :
- la reconstructibilité système
- la résilience aux redémarrages
- l’indépendance aux événements perdus

# ==========================================================
# 💡 AUTOMATION — ACTIVATION SOIR
# ==========================================================

## 🔄 Refonte de l’automation `1007000000003`

L’automation d’allumage soir :

- écrit explicitement les timestamps métier ;
- calcule et persiste les deadlines ;
- formalise l’idempotence du cycle ;
- documente les garanties causales.

---

## 🧠 Timestamps métier

Introduction de la notion :

> "commande d’allumage validée"

Le timestamp :
`input_datetime.jardin_soir_allumage_effectif`

représente désormais :
- l’émission validée de la commande ;
- et non une confirmation matérielle physique.

Cette distinction est désormais explicitement contractualisée.

---

## ♻️ Idempotence renforcée

L’allumage devient strictement :
- retrigger-safe ;
- restart-safe ;
- compatible concurrence événementielle.

Les timestamps ne peuvent plus être écrasés
par des retriggers concurrents.

# ==========================================================
# 🌙 AUTOMATION — EXTINCTION SOIR
# ==========================================================

## 🔄 Refonte complète de `1007000000004`

L’extinction soir est désormais pilotée :

- par deadlines persistantes ;
- via triggers temporels explicites ;
- sans dépendance à un sensor causal.

---

## ❌ Suppression du modèle causal précédent

L’automation ne dépend plus de :

`binary_sensor.lumiere_jardin_soir_extinction_autorisee`

comme source d’autorité.

Elle lit désormais directement :

- `input_datetime.jardin_soir_extinction_deadline`
- `input_datetime.jardin_sejour_inactivite_deadline`

---

## 🧠 Robustesse restart / reload

Ajout explicite de stratégies de rattrapage :

- `homeassistant.start`
- `automation_reloaded`

Garanties apportées :

- extinction post-redémarrage sans front perdu ;
- extinction post-reload ;
- reconstruction complète depuis les helpers persistants.

---

## 🚫 Interdits désormais formalisés

Le contrat d’extinction interdit explicitement :

- `time_pattern`
- `last_changed`
- `for:`
- causalité implicite
- substitution reload via `systeme_stable`

# ==========================================================
# 👁️ OBSERVABILITÉ ≠ CAUSALITÉ
# ==========================================================

## 🧠 Clarification doctrinale majeure

Le sensor :

`binary_sensor.lumiere_jardin_soir_extinction_autorisee`

devient officiellement :

- une projection déclarative ;
- un composant d’observabilité ;
- sans rôle causal.

---

## 🔍 Nouveau rôle du sensor

Le sensor sert désormais uniquement :

- à l’UI ;
- au diagnostic ;
- à l’observabilité métier ;
- à la cohérence documentaire.

Il n’est plus autorité système.

---

## ⚠️ Réévaluation implicite officiellement acceptée

Le contrat formalise désormais que :

- la réévaluation implicite via `now()`
  est acceptable pour l’observabilité ;
- mais interdite comme mécanisme causal.

Séparation désormais explicite :
- projection UI
vs
- moteur causal.
