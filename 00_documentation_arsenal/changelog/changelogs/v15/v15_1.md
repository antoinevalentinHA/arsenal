# Arsenal — Changelog v15.1

## Cadre

v15.1 est une version de consolidation opérationnelle.

Elle finalise deux chantiers ouverts autour de v15 :

1. clarification contractuelle de la projection MQTT de l’audit NAS vers `arsenal_self` ;
2. matérialisation du palmarès journalier de pluie dans Home Assistant.

---

## 1. Arsenal Self — Projection MQTT audit NAS

Le contrat MQTT de l’audit NAS passe en `v1.0.1`.

La responsabilité est clarifiée :

- le NAS produit `latest.verdict.json` ;
- `publish_audit_mqtt.sh` publie le verdict synthétique ;
- Home Assistant consomme officiellement via le domaine `arsenal_self` ;
- le détail patrimonial reste exclusivement dans le rapport NAS.

La projection MQTT transporte donc un verdict exploitable, pas le contenu complet de l’audit.

### Évolutions principales

- séparation plus stricte des autorités ;
- clarification du périmètre MQTT ;
- clarification des payloads d’erreur ;
- clarification du mode strict de fraîcheur ;
- extraction officielle du contrat Home Assistant `arsenal_self.md`.

Le contrat devient plus court, plus lisible et plus robuste doctrinalement.

---

## 2. Météo — Palmarès pluie journalier

Création du domaine fonctionnel de palmarès journalier de pluie.

Ajouts principaux :

- contrat `meteo/pluie_palmares.md` ;
- helpers de stockage des rangs 1 à 10 ;
- snapshot de la valeur journalière clôturée ;
- capteurs de synthèse et d’anomalie ;
- automatisations de snapshot et d’évaluation ;
- intégration Recorder ;
- création du `utility_meter` `sensor.pluie_journaliere` ;
- intégration Lovelace du palmarès historique.

L’ancienne note d’évolution future
`evolutions_futures/precipitations_records.md`
est supprimée : le chantier n’est plus théorique, il est implémenté.

---

## 3. Automatisations palmarès pluie

### Snapshot pré-minuit

Ajout de l’automatisation :

- `10160000000020`
- `Météo - Pluie - Palmarès journalier - Snapshot`

Rôle :

- capturer le cumul journalier avant reset du `utility_meter` ;
- isoler strictement la phase de clôture ;
- interdire toute logique de classement dans cette couche.

Déclenchement :

```text
23:59:55
```

---

### Évaluation du classement

Ajout de l’automatisation :

- `10160000000021`
- `Météo - Pluie - Palmarès journalier - Évaluation`

Rôle :

- évaluer le snapshot clôturé ;
- déterminer le rang d’insertion ;
- décaler les rangs historiques ;
- appliquer le top 10 ;
- horodater l’évaluation effective.

Le système refuse explicitement :

- snapshot obsolète ;
- anomalie structurelle ;
- doublon de date ;
- valeur nulle ;
- valeur hors classement.

L’automatisation applique un modèle FIFO déterministe de décalage des rangs.

## 5. Recorder & persistance

Le périmètre Recorder est étendu pour couvrir :

- les 10 rangs du palmarès ;
- les dates associées ;
- le snapshot veille ;
- l’horodatage d’évaluation ;
- les capteurs de synthèse et d’anomalie.

Objectif :

- persistance robuste ;
- restauration correcte après redémarrage ;
- cohérence historique du classement.

## 6. Lecture doctrinale

v15 avait rendu Arsenal observable par lui-même.

v15.1 ajoute un exemple propre de domaine métier local :

- source physique simple ;
- clôture temporelle explicite ;
- persistance contrôlée ;
- synthèse dédiée ;
- anomalie structurelle séparée ;
- UI alimentée par des entités métier ;
- absence de logique Lovelace cachée ;
- absence de dépendance Recorder comme source métier.

Le chantier pluie devient un exemple de pipeline métier Arsenal complet.

---

## 7. Pipeline release & gouvernance

v15.1 marque également une montée en maturité du pipeline de release NAS :

- séparation `digest` / `full diff` ;
- synthèse statistique ;
- gestion explicite des troncatures ;
- masquage volontaire des artefacts toxiques (`core.entity_registry`) ;

Le système produit désormais de véritables artefacts de release,
et non plus un simple dump de diff brut.
