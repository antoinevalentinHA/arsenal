## Lecture système

La release v15 fait passer Arsenal d'un système **observé de l'extérieur** à un système **qui s'observe lui-même**. C'est l'événement structurant de cette release : la mise en production de la brique `arsenal_self_audit`, jusqu'ici cantonnée à `evolutions_futures/moteur_audit.md`. Le moteur d'audit cesse d'être un projet et devient un module nominal du domaine `system`, avec sa chaîne complète : MQTT (cinq capteurs d'ingestion), template sensors (quatre capteurs dérivés dont l'alerte agrégée), helper de seuil de fraîcheur, carte Lovelace dédiée sur le dashboard `system`.

Le reste de la release relève de l'**hygiène architecturale** :

- Normalisation lexicale du slug `deltaT` → `deltat` sur toute la chaîne aération (contrats, script orchestrateur, template sensor de validation). Convergence avec la convention HA d'`entity_id` strictement minuscule. Aucun changement sémantique : ce sont les mêmes capteurs, désormais nommés conformément à la norme.
- Normalisation analogue sur l'UPS NAS Imprimerie : `binary_sensor.nas_imprimerie_ups_on_battery` → `binary_sensor.nas_imprimerie_ups_sur_batterie`. Cohérence francophone du domaine.
- Réorganisation hiérarchique : les sept `input_number.reload_integrations.*` migrent de `03_input_numbers/reload_integrations/` vers `03_input_numbers/system/reload_integrations/`. Pur déplacement, aucun renommage d'entité.
- Retrait de `input_boolean.bouclage_visiteur` du périmètre piloté par les modes maison (Normal et Vacances). Le helper n'est plus manipulé par les transitions de mode.

Côté documentation, la release acte deux migrations depuis `evolutions_futures/` vers `outils_externes/nas_arsenal/` :

- `moteur_audit.md` → `audit/audit.md` + `audit/mqtt.md` (suit la livraison du code).
- `nas_gestion_retention_sauvegardes.md` → `quarantine_purger.md` + `retention_manager.md` (livraison documentaire seule ; les briques NAS sont déjà en place côté Synology).

Aucun contrat normatif n'est créé ni modifié dans son fond. Aucune invariance système n'est levée. Aucun pipeline métier n'est restructuré.

## 1. Arsenal Self Audit (nouveau module système)

> **Statut** : nouveau module en production.
> **Origine** : `evolutions_futures/moteur_audit.md` (supprimé).
> **Destination doctrinaire** : `outils_externes/nas_arsenal/audit/`.

### 1.1 Chaîne d'ingestion MQTT

Cinq nouveaux capteurs MQTT exposent l'état du moteur d'audit exécuté côté NAS :

- `mqtt sensor.audit_published_at` — horodatage de la dernière publication d'audit.
- `mqtt sensor.audit_statut` — verdict synthétique du moteur.
- `mqtt sensor.audit_total_anomalies` — compteur d'anomalies relevées.
- `mqtt sensor.audit_total_observations` — compteur d'observations effectuées.
- `mqtt sensor.audit_version_auditee` — version d'Arsenal soumise à l'audit.

Ces capteurs sont **frontière de confiance** : Arsenal les ingère sans réinterprétation. Toute la décision métier est faite côté NAS.

### 1.2 Chaîne de dérivation locale (template sensors)

Quatre capteurs dérivés dans `12_template_sensors/system/audit/` :

- `sensor.audit_age_minutes` — âge de la dernière publication (dérivé de `audit_published_at`).
- `sensor.audit_stale` — capteur de fraîcheur, basé sur le seuil paramétrable.
- `sensor.audit_error` — détection d'état d'erreur de la chaîne MQTT.
- `sensor.audit_alerte` — agrégation des deux précédents en signal d'alerte unique.

La séparation `stale` / `error` / `alerte` respecte le pattern Arsenal de **capteurs métier composables** : chaque sous-condition est observable séparément, l'alerte est une dérivation pure.

### 1.3 Paramétrage utilisateur

- `input_number.audit_stale_threshold_hours` — seuil de tolérance de fraîcheur de l'audit. Helper paramétrique, sans valeur en dur dans le template.

### 1.4 Restitution UI

- `19_button_card_templates/.../arsenal_self_audit_status_card.yaml` — nouveau template de carte.
- Intégration sur le dashboard `system.yaml` immédiatement avant la section *Systèmes critiques*.

### 1.5 Lecture architecturale

Le module respecte la séparation tri-couche Perception / Décision / Exécution :

- **Perception** : MQTT (ingestion brute, NAS-side).
- **Décision** : template sensors (`stale`, `error`, `alerte`).
- **Exécution** : aucune. Le module est **strictement diagnostique**.

Aucune automatisation ne consomme `sensor.audit_alerte` à ce stade. Le module est en posture d'observation, conforme au pattern Arsenal pour les nouvelles briques diagnostiques (observation → recommandation → action, en trois temps).

Deux automations modifiées :

### 4.2 `11_automations/modes/vacances/application_debut.yaml`

Le helper sort du périmètre piloté par les modes maison. Il n'est pas supprimé du système (sa déclaration n'est pas dans le diff), mais sa **manipulation transitionnelle** est levée. Conformément à la doctrine Arsenal sur les helpers : un helper non manipulé par aucun consommateur devient dette technique candidate. Statut à arbitrer lors d'une future passe d'hygiène.

## Bilan des invariants

### Invariants renforcés en v15

- **Diagnostique d'abord, action ensuite** : la brique `arsenal_self_audit` est introduite en posture strictement observatoire. Aucun automatisme ne consomme `sensor.audit_alerte`. Conforme au pattern « observation → recommandation → action » en trois temps.
- **Séparation des frontières de confiance** : les capteurs `mqtt sensor.audit_*` sont des points d'entrée tracés ; toute la décision est isolée dans la couche template (`audit_stale`, `audit_error`, `audit_alerte`). Arsenal ne réinterprète jamais la donnée brute du NAS.
- **Paramétrage externalisé** : `input_number.audit_stale_threshold_hours` interdit toute valeur en dur dans les templates. Convention Arsenal sur tout seuil de décision.

### Invariants déjà établis et confirmés

- **Convention de slug `entity_id` minuscule strict** : appliquée rétroactivement à la chaîne ΔT et au binary_sensor UPS NAS Imprimerie. La convention n'est pas créée par v15, elle est consolidée.
- **Hiérarchie `03_input_numbers/system/...`** pour tout helper de gouvernance technique : appliquée aux sept helpers `reload_integrations`.
- **Séparation `unique_id` (immutable, registre) / `entity_id` (exposé, renommable)** : illustrée par le cas UPS — slug francisé, `unique_id` préservé.

### Invariants préservés (non touchés en v15)

- Domaine aération (chauffage / DeltaT / blocage) : aucune modification sémantique. M3 orchestrateur, capteurs ΔT, sensor de blocage valide → mêmes contrats, mêmes seuils, mêmes sources.
- Domaine modes maison : la transition Normal et l'application Vacances conservent toute leur séquence d'actions à l'exception du `bouclage_visiteur` (cf. §4).
- Contrats normatifs : aucun contrat de `00_documentation_arsenal/contrats/` n'est créé. Les deux modifications dans `aeration_blocage_chauffage/` portent exclusivement sur les slugs `deltaT` → `deltat`, pas sur la doctrine.
- Domaine NAS Imprimerie : la synthèse santé conserve son architecture en axes (réseau / VPN / stockage / UPS / uptime). Seul le slug du capteur UPS change.
