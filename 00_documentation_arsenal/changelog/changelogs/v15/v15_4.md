## 🎯 Synthèse exécutive

Release de **correction de pipeline temporel** et de **durcissement de l'automation de déclenchement d'alarme**. Trois corrections de fond convergent sur un même invariant : **un état booléen Home Assistant (`sun.sun`, `unavailable→off`) n'est pas une source de vérité fiable pour une décision métier ; seul un timestamp astronomique ou une transition contractualisée l'est.**

Une dette architecturale est par ailleurs documentée explicitement sur l'automation de déclenchement d'alarme (court-circuit du pipeline canonique Décision → Helpers → Application).

## 🔧 Corrections — Pipeline éclairage jardin matin

Le cycle matin reposait sur trois mécanismes fragiles indépendants qui pouvaient, dans certaines conditions saisonnières et de timing, produire :
- des allumages parasites pendant le cycle soir,
- une fenêtre matin active alors que le soleil est déjà levé,
- une heure d'extinction ancrée sur le lever du **lendemain** au lieu du jour courant.

Les trois fichiers concernés sont durcis simultanément avec une logique de sélection du lever astronomique commune.

### `binary_sensor.jardin_cycle_matin_actif` (template, v1.2)

**Remplace** `is_state('sun.sun', 'below_horizon')` par une comparaison directe `now() < lever_astronomique`.

**Motif** : l'état `sun.sun` peut rester `below_horizon` plusieurs minutes après le lever réel (latence de mise à jour HA). Le timestamp astronomique (`next_rising` / `last_rising`) est déterministe.

**Logique canonique de sélection du lever du jour courant** (réutilisée en aval) :
1. `next_rising` si sa date == aujourd'hui (on est avant le lever)
2. `last_rising` si sa date == aujourd'hui (on est après le lever)
3. Sinon → `none` → fenêtre inactive par sécurité

**Ajoute** : garde `h_fin > h_debut` (fenêtre structurellement non vide).
**Motif** : en mai à Bordeaux, le lever du soleil précède l'heure d'allumage configurée → fenêtre vide → aucun allumage parasite.

### `automation.maj_heure_extinction_jardin_matin` (v1.2)

**Remplace** le trigger `platform: sun / event: sunrise` par `platform: time / at: "00:01:00"`.

**Motif** : déclencher au lever du soleil crée une race condition (`dt ≈ now_ts` à quelques millisecondes près) rendant la garde `dt < now_ts` non déterministe. Le recalcul à 00:01 utilise `next_rising` du jour, qui est connu et strictement futur à cette heure.

**Supprime** le bloc de correction `+ timedelta(days=1)`.
**Motif** : `input_datetime` ne stocke que l'heure (`%H:%M:%S`), la correction "lendemain" était sans effet réel et masquait une logique conditionnelle cassée.

**Aligne** la sélection du lever sur la logique canonique du binary sensor, avec fallback sur l'heure fixe seule.
**Motif** : si un helper est modifié après le lever, `next_rising` pointe déjà sur demain. Sans cette garde, le calcul serait ancré sur le lever du **lendemain**, produisant une heure d'extinction contractuellement fausse.

### `automation.extinction_jardin_matin` (v1.1)

**Trigger** : ajout de `from: 'on'` sur la transition d'état.
**Motif** : sans `from: 'on'`, la transition `unavailable → off` (rechargement de templates HA) déclenchait une extinction parasite, y compris pendant le cycle soir. Seule la transition `on → off` est sémantiquement significative.

**Condition** : ajout d'une garde sur `binary_sensor.jardin_cycle_eclairage_soir_actif == 'off'`.
**Motif** : conforme au **contrat §4 (règle d'arbitrage unique)** — la politique SOIR est prioritaire sur la politique MATIN. L'extinction matin ne doit **jamais** s'exécuter si le cycle soir est actif, quelle qu'en soit la cause technique.

## 🚨 Durcissement — Alarme intrusion (délai d'entrée)

### `automation.alarme_delai_entree_fin` (10020000000032)

Ajout de **deux gardes** sur le déclenchement de l'alarme à expiration du timer d'entrée :

1. `input_boolean.systeme_stable == 'on'` — garde **post-reboot** : l'alarme ne peut pas se déclencher tant que le système n'a pas confirmé sa stabilisation après redémarrage.
2. `binary_sensor.ouverture_qualifiee_maison == 'on'` — garde **intrusion toujours active** : l'alarme ne se déclenche que si l'ouverture qualifiée qui a armé le délai d'entrée est encore présente au moment de l'expiration.

### ⚠️ Dette architecturale documentée

L'automation `10020000000032` **court-circuite** le pipeline canonique Arsenal :

> Décision → Helpers (`input_text.alarme_etat_cible`) → Application (`script.alarme_decision_centrale`)

Elle appelle directement `alarm_control_panel.alarm_trigger` et `script.sirene_brutale`. Cette dette est désormais **explicitement consignée dans l'en-tête du fichier**.

**Refonte cible** (non incluse dans v15.4) :
- Matérialiser **"intrusion confirmée"** comme état contractuel persisté.
- Déclencher panneau d'alarme et sirène depuis cet état via le pipeline canonique.
