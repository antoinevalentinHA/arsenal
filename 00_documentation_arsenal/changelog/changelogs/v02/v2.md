## Préambule (cadre G2)
Cette transition est un **durcissement ciblé** : elle renforce surtout le couple **Aération ↔ Chauffage** (blocage + récupération thermique) et clarifie la sémantique des **fenêtres “avec délai”** (séjour/parents/étage/maison), avec un gain net d’**observabilité** (capteurs diagnostic chauffage + timers affichés).

---

## 📌 Points majeurs (signal)

### 1) Aération → Blocage chauffage + récupération thermique (chantier structurant)
Ajouts structurants :
- Nouveau **bloc “blocage_chauffage”** (plusieurs automations : blocage / chute température / début & fin de récupération / réactivation / sécurité / sauvegarde du temps de récupération).
- Nouveaux artefacts d’état/mesure :
  - `input_number` dédié au **temps de récupération thermique**
  - `input_boolean` “reprise thermique en cours”
  - `template sensors` “début/fin/temps” de récupération thermique

Lecture “signal” :
- Arsenal introduit une **machine d’état explicite** autour de “l’après-aération” (récupération thermique), au lieu d’un comportement implicite.
- On bascule vers une gouvernance “mesure → décision → actions”, plus robuste et plus auditable.

Risques :
- Effet domino si un des éléments manquants (timer/helper/sensor) : récupération thermique muette ou incohérente.
- Si la “chute de température” est mal calibrée : blocage trop long ou trop court → inconfort / surconsommation.

---

### 2) Chauffage → introduction d’un cerveau “décision centrale” + diagnostic visible
Ajouts clés :
- Script `chauffage_decision_centrale` (nouveau pilier : une autorité de décision explicite).
- Automation de déclenchement dédiée.
- Nouveaux capteurs diagnostic :
  - `sensor.chauffage_mode_calcule`
  - `sensor.chauffage_raison_calculee`

Et côté UI (diagnostics chauffage) :
- Affichage “mode calculé / raison calculée” + états des timers d’antirebond.

Lecture “signal” :
- On arrête de piloter “à l’aveugle” : Arsenal expose la **raison** et le **mode** calculés, et rend visibles les garde-fous (antirebond).
- C’est un pas net vers une **architecture gouvernée** (décision centrale + observabilité).

Risques :
- Si la décision centrale devient un passage obligé, toute régression dans ce script devient systémique (single point of failure).
- Les diagnostics UI créent une attente : si un capteur est `unknown`, l’interprétation humaine devient fragile.

---

### 3) Ouvertures “avec délai” — changement de sémantique (point critique)
Changements majeurs dans les templates “avec délai” :
- Avant : logique centrée sur “timer actif = ON”.
- Après : logique centrée sur “fenêtre ouverte ET délai expiré (timer = idle) = ON”.
- Extension :
  - ajout d’un capteur “étage avec délai” intégré dans le global maison
  - “étage avec délai” devient “enfants immédiat OU parents ouverts + timer parents idle”

Lecture “signal” (très important) :
- Ces capteurs ne représentent plus “un délai en cours”, mais **une ouverture persistante au-delà du délai de grâce**.
- C’est exactement ce qu’il faut pour du chauffage : ne pas couper au premier souffle d’ouverture, mais agir si ça dure.

Risques :
- **Risque sémantique** : les noms restent “avec délai / grâce” → un lecteur peut croire que ON signifie “délai en cours”, alors que c’est l’inverse (délai expiré + ouverture toujours présente).
- Effet fonctionnel : tout ce qui consomme ces sensors doit être aligné (sinon décisions inversées).

---

### 4) Timers “ouvertures grâce” — simplification assumée (mais attention)
Suppression du timer “grâce étage” (le système n’en dépend plus explicitement dans les templates).
Conséquence logique :
- L’étage s’appuie désormais sur :
  - ouverture enfants (immédiat)
  - ouverture parents + timer parents idle (donc “grâce expirée”)

Risques :
- Si tu attendais une logique de “grâce étage” autonome (séparée des parents), elle n’existe plus : c’est un choix de modèle.

## 🧾 Added / Removed net (signal utile)

### Ajouts (signal)
- Bloc complet “aération → blocage chauffage → récupération thermique”
- Décision centrale chauffage + capteurs de diagnostic
- Nouvelle automation “fenêtres séjour” (surface fonctionnelle accrue côté ouvertures)

### Suppressions (signal)
- Débranchement de l’ancien sous-système “fenêtres chauffage” (plusieurs automations Vicare fenêtres, sécurité, timers, réactivation…)
- Suppression de l’ancienne geolocalisation chauffage (au profit d’une orchestration plus centralisée)
- Suppression d’un capteur “recovery” chauffage (remplacé par un modèle plus explicite/structuré)

Lecture :
- Ce n’est pas une simple optimisation : c’est un **remplacement de stratégie** sur “fenêtres ↔ chauffage”.

Risques :
- Si la décision centrale ne couvre pas 100% des cas auparavant gérés par l’ancien paquet “fenêtres”, tu peux perdre des protections implicites (anti-blocage, réactivation auto, etc.).

## 🧱 CONFORMITÉ AUX PRINCIPES ARSENAL

### Robustesse / anti-silence
- Introduction de signaux explicites (capteurs diagnostic chauffage, timers de grâce interprétés clairement côté UI).
- Remplacement de mécanismes implicites par une machine d’état (récupération thermique).

### Gouvernance / lisibilité
- Reclassement “aération/conseil” + restructuration des notifications.
- Clarification “avec délai” (même si le naming doit être surveillé pour éviter l’inversion mentale).

### Observabilité
- Diagnostics chauffage enrichis (mode/raison, antirebond).
- Diagnostics ouvertures structurés : instantané vs “avec délai”.

---

## 🚫 Bruit identifié (dépriorisable)
- Réécriture “cosmétique” d’en-têtes/commentaires (notamment automation fenêtre parents) : le fond change peu, la forme beaucoup.
- Ajustements UI de présentation (emoji, sections) : utile pour lecture, mais non structurant.

---

## ✅ ÉTAT FINAL (G2)
- Objectif atteint : **durcissement ciblé** Aération↔Chauffage + meilleure observabilité chauffage + sémantique “avec délai” alignée sur un usage chauffage.
- Dette technique ajoutée : **faible**, principalement un risque de **confusion sémantique** (“grâce/avec délai”) si la doctrine n’est pas explicitée dans la doc.
- Risque principal : régression si l’ancien paquet “fenêtres↔chauffage” retiré n’est pas totalement couvert par la décision centrale + le nouveau bloc aération.
