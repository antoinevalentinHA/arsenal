==============================
🧠 ARSENAL HA — v6.0 STABLE
==============================

Architecture domestique automatisée fondée sur :
- des états métier explicites,
- des décisions centralisées,
- une séparation stricte backend / UI,
- une robustesse totale aux redémarrages et aux APIs instables.

Cette version constitue le **socle restaurable V6** : modèle temporel unifié (ECS-like), UI gouvernée, décisions souveraines, et observabilité maîtrisée.

---

## 🧱 1. ARCHITECTURE FONDATRICE ARSENAL

### Séparation stricte des responsabilités
- Backend :
  - décide, arbitre, applique
- UI :
  - lit, explique, permet l’action humaine explicite
- Aucune logique métier dans Lovelace
- Aucun seuil métier codé dans l’UI

### Décision centrale souveraine
- Chaque domaine critique (chauffage, clim, ECS, présence, éclairage…) possède :
  - un cerveau unique
  - des vérités métier explicites
- Les automatisations ne se pilotent jamais entre elles
- Les modes globaux (Vacances, blocages, tests) n’agissent que via des états métier

### Redémarrage = cas nominal
- Aucun système ne dépend :
  - de timers fragiles
  - d’états transitoires non recalculables
- Recalculabilité et déterminisme comme contraintes de base

---

## ⏱️ 2. MODÈLE TEMPOREL UNIFIÉ (ECS-LIKE)

### Principe cardinal
> Le temps est une DONNÉE, jamais un trigger permanent.

### Abandons structurants
- polling (`time_pattern`)
- timers fragiles
- comparaisons exactes `HH:MM`
- `input_datetime` dynamiques comme sources de décision
- pilotage ON/OFF basé sur scans périodiques

### Adoption du modèle stateless
- Les fenêtres temporelles sont matérialisées par des **vérités métier** (binary_sensors)
- Les automations deviennent :
  - événementielles
  - déclenchées par transitions d’état
  - idempotentes et reboot-safe

Applications :
- ECS ponctuel & désinfection
- simulation de présence
- logiques horaires clim / chauffage
- éclairage extérieur (jardin matin / soir)

## 🔒 11. ROBUSTESSE AUX APIs EXTERNES (WITHINGS)

  - dérivés boot-safe
