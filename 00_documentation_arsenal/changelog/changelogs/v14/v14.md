# Arsenal — Changelog v14

**Statut** : majeure structurelle.
**Portée** : refonte de l’arborescence racine, formalisation d’un corpus de contrats normatifs, refonte du domaine **Voiture / Autonomie**, intégration **Cumulus petite maison** dans la chaîne d’observabilité.

---

## 1. Invariants nouveaux

### 1.1 Nomenclature canonique des includes (refonte structurelle)

L’arborescence racine adopte un **préfixe numérique strict** pour tous les dossiers consommés par `configuration.yaml`. La séquence d’inclusion devient un **ordre de chargement explicite et auditable**.

| Avant (v13.3) | Après (v14) |
|---|---|
| `02_groups` … `08_timers` | inchangé |
| `08b_counters` | `09_counters` |
| `09_scripts` | `10_scripts` |
| `10_automations` | `11_automations` |
| `11_template_sensors` | `12_template_sensors` |
| `12_sensor_platforms` | `13_sensor_platforms` |
| `mqtt_sensors` | `14_mqtt_sensors` |
| `mqtt_binary_sensors` | `15_mqtt_binary_sensors` |
| `template_alarm_panels` | `16_template_alarm_panels` |
| `zones` | `17_zones` |
| `lovelace` | `18_lovelace` |
| `button_card_templates` | `19_button_card_templates` |
| `documentation_arsenal` | `00_documentation_arsenal` |

**Conséquences contractuelles** :

- Le numéro de préfixe **n’est pas décoratif** : il fixe l’ordre canonique de lecture (helpers → timers → counters → scripts → automations → templates → sensors → MQTT → panels → zones → UI).
- La documentation (`00_*`) précède strictement le code, ce qui rend visible la primauté du contrat sur l’implémentation.
- Tous les en-têtes (`# 🔗 Utilisé par …`) des fichiers helpers, scripts, automatisations, templates ont été mis en cohérence avec la nouvelle nomenclature. Aucune logique métier n’a bougé dans ces ré-écritures d’en-têtes.

### 1.2 Corpus de contrats normatifs `00_documentation_arsenal/contrats/`

Le dossier `contrats/` devient le **point d’entrée unique** de la gouvernance Arsenal. Les domaines suivants disposent désormais d’un contrat publié, opposable et versionné :

- **Alarme** — gouvernance complète éclatée en 13 fichiers numérotés (`00_gouvernance_alarme.md`, `10_modele_etats_et_vocabulaire.md`, `20_interfaces_contexte_et_helpers.md`, `30_decision_centrale.md`, `40_application_decision.md`, `50_intrusion_detection.md`, `51_ouvrants_entree.md`, `60_delais_et_blocages.md`, `61_watchdog_blocage_armement.md`, `70_sirene_actions_terminales.md`, `80_notifications_et_feedback.md`, `90_ui.md`, `95_diagnostics_et_coherence.md`, `99_hors_perimetre_et_extensions.md`).
- **Chauffage** — gouvernance et capteurs en 13 fichiers (`00_gouvernance_chauffage.md`, `10_souverainete_execution.md`, `15_capteurs/01_capteurs_decision.md`, `15_capteurs/03_capteurs_blocages_niveau1/fenetre_ouverte_maison_avec_delai.md`, `20_triggers_decisionnels.md`, `45_aeration.md`, `46_aeration_observation_thermique.md`, `50_standby_hysteresis.md`, `60_absence_inhibition_geofencing.md`, `65_pre_confort_retour_vacances.md`, `70_autorisation_thermostat.md`, `75_auto_ajustement_courbe.md`, `90_semantique_thermique.md`).
- **ECS** — fondations, principes, gouvernance, orchestration (`00_fondations_et_statut.md`, `01_principes_perimetre_et_roles.md`, `02_gouvernance_autorites_et_chaine.md`, `03_orchestration_et_wrappers.md`, …).
- **Aeration / blocage chauffage** — socle transversal (`socle_transversal/01__objet_perimetre_statut.md`).
- **Boiler** — `boiler/README.md` formalisant le contrat MQTT du bridge.
- **Cumulus petite maison** — `cumulus_petite_maison.md`.
- **Bouclage**, **Babysitting** — contrats dédiés.

**Principe** : aucun YAML d’un domaine couvert ne doit plus exister sans contrat amont. Toute évolution future passe d’abord par le contrat.

### 1.3 Domaine Voiture / Autonomie — passage à un snapshot atomique

`03_input_numbers/voiture/autonomie.yaml` passe d’un helper unique à un **trio de helpers cohérents**, écrits dans la même transaction logique (snapshot pleine charge ≈ 100 %) :

- `autonomie_audi_etron_full` — autonomie électrique réelle (km), conservé.
- `audi_temperature_charge` — température extérieure au moment de la pleine charge (°C, [-20, 50], step 0.1) — **nouveau**.
- `audi_autonomie_corrigee_temperature` — autonomie normalisée à 20 °C (km, step 0.1) — **nouveau**.

**Invariant critique posé en en-tête** :

> Tous les helpers de ce fichier doivent être écrits dans la MÊME automation et au MÊME instant logique (snapshot pleine charge).

Conséquence : la valeur figée n’est plus une grandeur isolée, elle devient un **tuple cohérent** permettant l’analyse de l’influence thermique sur l’autonomie.

L’ancienne automation `10_automations/voiture/autonomie.yaml` est supprimée et remplacée par sa version sous `11_automations/voiture/autonomie.yaml`, refondue pour produire ce snapshot atomique.

### 1.4 Cumulus petite maison — entrée dans l’observabilité

`recorder.yaml` ouvre une nouvelle catégorie « **CONSOMMATION – CUMULUS PETITE MAISON** » avec inclusion explicite de `sensor.petite_maison_modbuslink_1_2_electric_energy_consumption`. Le cumulus du site distant entre dans le périmètre des données conservées par le recorder, en cohérence avec le contrat dédié (`contrats/cumulus_petite_maison.md`).

Ajouts symétriques dans la catégorie véhicule : `input_number.audi_temperature_charge`, `input_number.audi_autonomie_corrigee_temperature`, `sensor.audi_temperature_charge_mensuelle`.

## 5. Migration

Toute installation passant de v13.3 à v14 doit :

1. Renommer les dossiers racine selon le tableau §1.1 **avant** de redémarrer Home Assistant.
2. Vérifier que `configuration.yaml` reflète exactement la nouvelle nomenclature (incluant `13_sensor_platforms/`, `14_mqtt_sensors/`, `15_mqtt_binary_sensors/`, `16_template_alarm_panels/`, `17_zones/`, `18_lovelace/`).
3. Relire les contrats nouvellement publiés sous `00_documentation_arsenal/contrats/` avant toute modification d’un domaine couvert (alarme, chauffage, ECS, boiler, bouclage, aération, babysitting, cumulus petite maison).
4. Pour le domaine voiture : valider que l’automation `11_automations/voiture/autonomie.yaml` écrit bien les **trois** helpers en transaction unique. Une écriture partielle viole l’invariant §1.3.
5. Si les domaines suivants étaient utilisés, prévoir leur réimplémentation hors v14 : ouvertures redondantes, consolidation sommeil, sensors transactionnels boiler.
