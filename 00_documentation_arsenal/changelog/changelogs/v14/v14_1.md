# Arsenal — Changelog v14.1

## Cadre

v14.1 est une **version de mise en ordre**.

Aucun comportement métier n'évolue. Aucune décision n'est déplacée, aucun pipeline n'est modifié, aucun seuil n'est retouché. La version consolide trois chantiers strictement structurels :

1. la **réorganisation de la documentation d'architecture** (numérotation, regroupement par axe) ;
2. l'**extraction de domaines transverses** dans `01_customize/` et `02_groups/` (connectivité, proxys, météo) ;
3. la **normalisation des en-têtes** sur l'ensemble des helpers et groupes touchés par la convention Arsenal v14.

Un seul périmètre fonctionnel apparaît : la **synthèse Ping LAN**, formalisée comme domaine de supervision réseau dédié, avec contrat, groupes, capteur de synthèse et carte de diagnostic.

## 1. Réorganisation de la documentation d'architecture

### 1.1 Numérotation des axes

`00_documentation_arsenal/architecture/` adopte une numérotation explicite reflétant la dépendance de lecture :

- `00_structure_includes/` — inventaire des includes YAML (nature, périmètre, interdits par dossier)
- `00_system_log.md` — politique du System Log
- `01_logger.md` — politique du logger
- `01_recorder/` — contrat et fiche de décision du Recorder
- `02_etiquettes/` — conventions d'étiquetage par famille (`automatisations`, `capteurs`, `helpers`, `scripts`)
- `02_logbook.md` — politique du Logbook

Les anciens emplacements plats (`architecture/etiquettes/…`, `architecture/logger.md`, `architecture/logbook.md`, `architecture/recorder/…`, `architecture/structure_includes/…`, `architecture/system_log.md`) sont supprimés. Les contenus correspondants sont migrés sans modification de fond ; seul l'ordonnancement change.

### 1.2 `00_structure_includes/` — un document par dossier YAML

Le dossier remplace l'ancien fichier monolithique `structure_includes.md` et l'arborescence plate `structure_includes/`. Chaque dossier de configuration dispose désormais d'une fiche dédiée :

`00_structure_includes.md`, `01_customize.md`, `02_groups.md`, `03_input_numbers.md`, `04_input_texts.md`, `05_input_booleans.md`, `06_input_selects.md`, `07_input_datetimes.md`, `08_timers.md`, `09_counters.md`, `10_scripts.md`, `11_automations.md`, `12_template_sensors.md`, `13_sensor_platforms.md`, `14_mqtt_sensors.md`, `15_mqtt_binary_sensors.md`, `17_zones.md`, `18_lovelace.md`, `button_card_templates.md`, `logbook.md`, `logger.md`, `recorder.md`, `utility_meter.md`.

Chaque fiche définit le **rôle**, le **périmètre**, les **interdits**, et le **statut** du dossier qu'elle gouverne. Aucun fichier YAML n'est admis dans un dossier sans en-tête conforme à la fiche correspondante.

### 1.3 Nouveau document : `architecture/git.md`

Formalise la politique git du dépôt Arsenal (structure, conventions de commit, intégration au workflow cognitif). Ce document n'avait pas d'équivalent en v14 ; le sujet était jusqu'ici implicite.

### 1.4 Évolutions futures

Le dossier `evolutions_futures/` est mis à jour :

- **Ajouté** : `cumulus_petite_maison.md`, `lovelace_arborescence.md`, `precipitations_records.md`
- **Retiré** : `conso_cumulus.md` (remplacé par `cumulus_petite_maison.md`, périmètre élargi)

Aucun de ces documents n'est implémenté. Ce sont des **fiches d'intention** au sens Arsenal : périmètre cadré, décisions ouvertes, pas de code associé.

### 1.5 Changelog v14 archivé

`changelog/changelogs/v14/v14.md` est ajouté au dépôt. Le présent document succède à celui-ci.

## 3. Domaine Ping LAN — supervision réseau

C'est le **seul périmètre fonctionnel ajouté** en v14.1.

### 3.1 Contrat

`00_documentation_arsenal/contrats/ping_lan_synthese.md` formalise la supervision ICMP du LAN comme un domaine à part entière, distinct du domaine `connectivite/` (qui mesure la qualité de lien sans-fil par technologie).

### 3.2 Stratification des hôtes

`02_groups/ping/` introduit six groupes de criticité :

- `lan_critique.yaml` — équipements dont l'indisponibilité dégrade immédiatement Arsenal
- `lan_importante.yaml` — équipements dont l'indisponibilité dégrade un domaine métier
- `lan_secondaire.yaml` — équipements de confort
- `lan_esp32.yaml` — ESP32 (BLE proxy SwitchBot, etc.)
- `lan_stations_meteo.yaml` — stations météo Netatmo / consoles
- `lan_hosts.yaml` — agrégat de l'ensemble (utilisé en lecture par la synthèse)

L'invariant : **chaque hôte LAN appartient à exactement un groupe de criticité** ; le groupe `lan_hosts` est lecture seule et ne porte aucune décision.

### 3.3 Synthèse

`12_template_sensors/system/connectivite/ping_lan_synthese.yaml` produit un capteur de synthèse `sensor.ping_lan_synthese` consommé par le tableau de diagnostic. Le capteur est strictement lecture — aucune décision automatique n'est attachée à sa valeur en v14.1.

### 3.4 Carte de diagnostic

`19_button_card_templates/40_dashboards/system/30_diagnostic_connectivite/ping_lan_synthese.yaml` ajoute le rendu Lovelace de la synthèse au dashboard de diagnostic connectivité.

## 4. Normalisation des en-têtes — convention v14

Tous les fichiers touchés par v14.1 adoptent la **grammaire d'en-tête canonique Arsenal v14**. Ce chantier représente l'essentiel du volume du diff.

### 4.1 Sections canoniques

L'en-tête de tout fichier YAML Arsenal est désormais structuré ainsi :

```
# 🧠 ARSENAL — <FAMILLE>
#     <Domaine> — <Sous-objet>
# ----------------------------------------------------------
# 🎯 RÔLE
#   <une à trois lignes décrivant ce que le fichier fait>
#
# 🧩 PÉRIMÈTRE
#   - <inclusion 1>
#   - <inclusion 2>
#   - <interdit implicite formulé en positif>
#   - Aucune logique métier
#   - Aucun calcul d'état implicite
#   - Aucun comportement dynamique
#
# 🏷️ NATURE                       (helpers uniquement)
#   <Paramètre utilisateur | Seuil métier | Mémoire | …>
#
# 🚫 INTERDITS
#   - <interdit 1 formulé en impératif négatif>
#   - …
#
# 🏷️ STATUT
#   Socle — Arsenal v14.x
# ==========================================================
```

### 4.2 Sections supprimées

Les en-têtes hérités des versions antérieures contenaient des sections dont la présence dans le fichier YAML était devenue redondante avec les contrats :

- **Hiérarchie décisionnelle** — relevait du contrat de domaine, pas du helper individuel
- **Lecture métier** — paraphrasait le rôle
- **Dépendances → Utilisé par** — listait des consommateurs (ex. `11_automations/imprimerie/atelier/min_max.yaml`) ; cette information vit désormais uniquement dans le contrat ou dans le graphe de références produit par l'audit Python. Un helper n'a pas à connaître ses consommateurs.
- **Utilisation** — doublon de `Utilisé par`
- **Objectif** — paraphrasait le rôle
- **Unité** — dupliquait `unit_of_measurement` déjà présent dans la définition

### 4.3 Casse et ponctuation

- Les titres de sections passent en **majuscules** (`🎯 RÔLE` au lieu de `🎯 Rôle :`).
- Les apostrophes ASCII (`'`) sont normalisées en apostrophes typographiques (`'`).
- Le statut migre de `Arsenal v6.x — socle…` (mention historique) à `Socle — Arsenal v14.x` (mention de version courante de la convention d'en-tête).

### 4.4 Périmètre touché

Headers normalisés en v14.1 :

- `03_input_numbers/` — 94 fichiers
- `04_input_texts/` — 6 fichiers
- `13_sensor_platforms/` — 46 fichiers (modifications strictement d'en-tête)
- `02_groups/` — 11 fichiers (intégrations + petite_maison_cumulus + switchbot)
- `09_counters/` — 4 fichiers
- `15_mqtt_binary_sensors/` — 3 fichiers
- `14_mqtt_sensors/` — 11 fichiers
- `18_lovelace/` — 8 fichiers
- `11_automations/` — 2 fichiers
- divers (zigbee2mqtt, storage, racine) — résiduels

**Aucun de ces fichiers ne change de comportement.** Le diff sur ces fichiers est strictement éditorial.

## 5. Lovelace — `graph_cardio_nuit`

Modifications fonctionnelles sur la carte cardio nuit :

- série `FC nuit` (colonne) : couleur explicite `rgba(76, 175, 80, 0.85)` (vert)
- série `Baseline 7j` (ligne) : couleur explicite `rgba(144, 202, 249, 0.9)` (bleu clair)
- série `Baseline 7j` : épaisseur de trait `stroke_width: 2` → `3`

L'objectif est de rendre la baseline lisible quand elle se superpose à la série de barres, sans modifier le calcul ni la fenêtre temporelle (15 j inchangés).

Le fichier subit également une normalisation de fins de ligne (CRLF → LF). Cette part du diff est sans impact runtime.

## 6. Ce que v14.1 n'est pas

Pour mémoire, et pour éviter toute confusion à la lecture du diff brut :

- Aucun seuil métier n'est modifié (chauffage, ECS, climatisation, VMC, déshumidificateur, aération).
- Aucun pipeline décisionnel n'évolue.
- Aucune nouvelle automatisation décisionnelle n'est introduite. Les deux `11_automations/` modifiées sont strictement éditoriales.
- Aucune entité n'est renommée. Tous les renommages de v14.1 portent sur des **fichiers** YAML, pas sur des **entités** Home Assistant.
- Le volume `git/` listé dans le diff (`+347 / -11264`) est artefact du repository git lui-même (objets de packfile) et n'a pas de signification métier.

## 8. Migration

Aucune action utilisateur. Aucun reload sélectif n'est nécessaire au-delà du redémarrage habituel post-mise-à-jour. Aucune persistance n'est invalidée. Aucune entité n'est dépréciée.

Les chemins ayant changé en v14.1 sont des chemins de **fichiers de configuration** ; ils sont transparents pour Home Assistant dès lors que le `packages:` racine référence les nouveaux dossiers. Vérifier une fois après mise à jour que `01_customize/connectivite/`, `01_customize/proxys/`, `01_customize/meteo/` et `02_groups/connectivite/` sont bien chargés (présence des entités attendues dans le registre).
