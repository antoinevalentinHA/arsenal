# Arsenal — Changelog v13.3

> **Périmètre.** Introduction du sous-système **Cardio nuit** (pipeline complet snapshot → baseline → état → anomalie → confirmation), bascule du modèle **couleur respiratoire** vers une référence personnelle 14 j, durcissement de la consolidation **Sommeil** par triggers temporels de rattrapage, correction des entités **volets pluie** sur le nommage canonique `cover.<piece>`, et structuration documentaire (`etiquettes/`, `recorder/`).

---

## Lecture stratégique

```text
v13.2 → système auditable
v13.3 → système physiologique interprété
```

Trois mouvements convergents :

1. **Extension du pattern « snapshot + interprétation » au domaine cardio.** Le mouvement appliqué au sommeil en v13.2 (découplage des entités Withings, snapshot Arsenal consolidé, capteurs dérivés, dashboard dédié) est répliqué intégralement sur la fréquence cardiaque nocturne — avec une couche supplémentaire : la **confirmation de persistance** d'anomalie.
2. **Abandon des seuils absolus universels au profit d'une référence personnelle.** Le pipeline couleur respiratoire bascule sur `delta vs moyenne 14 j`. Cohérence inter-domaines avec cardio.
3. **Sécurisation de la consolidation sommeil.** Ajout de triggers temporels de rattrapage en complément du trigger événementiel.

Le saut sémantique :

> v13.2 exposait l'état du système.
> v13.3 expose un **état physiologique interprété**.

---

## ⚠️ Point d'attention (auto-challenge)

Le passage à des seuils relatifs (delta vs baseline) introduit un risque latent : la **dérive lente de la baseline elle-même**. Si la condition physiologique de référence se dégrade progressivement, la baseline glissante 14 j absorbe la dérive et les couleurs restent vertes. Ce point devra être instruit en v13.4+ par une couche de surveillance sur la baseline elle-même (référence longue durée vs glissante 14 j).

### 1.1. Architecture du pipeline

```text
mesure Withings
  → snapshot Arsenal (input_number/text/boolean/datetime)
  → état qualitatif + delta baseline

  → dashboard dédié + carte synthèse
```

### 1.4. Pipeline couleur — Réécriture complète

`12_template_sensors/couleurs/sante/frequence_cardiaque.yaml` ne lit plus `sensor.withings_average_heart_rate_local`. Trois capteurs couleur exposés au socle UI :

| Capteur | Source | Sorties |
|---|---|---|
| `couleur_cardio_nuit_etat` | `cardio_nuit_etat` + `binary_sensor.cardio_anomalie.confirmation` | `blue` / `green` / `yellow` / `orange` / `red` / `unavailable` |
| `couleur_cardio_nuit_delta_baseline` | `cardio_nuit_delta_baseline` | `blue` / `green` / `yellow` / `orange` / `unavailable` |
| `couleur_cardio_nuit_anomalie` | `cardio_anomalie` (état + `confirmation`) | `grey` / `yellow` / `red` / `unavailable` |

**Sémantique de la confirmation.** `couleur_cardio_nuit_etat` distingue `anormal` non confirmé (`orange`) de `anormal` confirmé (`red`). Idem pour `couleur_cardio_nuit_anomalie` : `on` non confirmé → `yellow`, `on` confirmé → `red`. Lecture via `state_attr('binary_sensor.cardio_anomalie', 'confirmation')` avec garde de cast (`true` ou chaîne `'true'` insensible à la casse).

## 2. Couleur respiratoire — Bascule vers référence personnelle

### 2.1. Modèle de seuils

| Aspect | Avant (v13.2) | Après (v13.3) |
|---|---|---|
| Référence | Seuils absolus universels (`11` / `17` / `20` / `23` rpm) | Moyenne glissante 14 j personnelle |
| Plage de validité | Implicite | `[8, 30]` rpm explicite (`rpm_min` / `rpm_max`) |
| Calcul | Comparaison directe `rpm < seuil` | Delta `rpm − ref_14j` puis seuils sur le delta |
| Indispo source | `unavailable` | `unavailable` |
| Indispo référence | n/a | `unavailable` (garde explicite) |
| Hors plage | n/a | `unavailable` (source ou référence) |

Seuils sur le delta :

| Delta (rpm) | Couleur |
|---|---|
| `≤ −1.5` | `blue` |
| `≤ +1.0` | `green` |
| `≤ +2.0` | `yellow` |
| `≤ +3.0` | `orange` |
| `> +3.0` | `red` |

### 2.2. Attributs diagnostic

Le capteur expose désormais des attributs d'audit UI :

- `reason` — énumération priorisée (`source_unavailable`, `reference_unavailable`, `source_out_of_range`, `reference_out_of_range`, `ok`)
- `rpm` — valeur source brute
- `reference_7j` / `reference_14j` / `reference_30j` — exposées pour audit, **seule `reference_14j` participe à la décision en v1**
- `delta_14j` — delta numérique arrondi à 1 décimale

## 3. Sommeil — Consolidation événementielle + rattrapage temporel

### 3.1. Triggers de l'automation `sommeil_consolidation.yaml`

| Avant (v13.2) | Après (v13.3) |
|---|---|
| `state` sur `binary_sensor.sommeil_donnees_exploitables` (à `on` 5 min) | Idem + `time: 06:00:00` + `time: 09:00:00` |

```text
événementiel pur → événementiel + rattrapage temporel
```

Les conditions horaires existantes (`after: 06:00:00`, `before: 11:00:00`) restent inchangées : les triggers temporels ne déclenchent qu'à l'intérieur de la fenêtre opératoire. L'ajout sécurise la consolidation lorsque la transition de `donnees_exploitables` survient avant le démarrage de HA ou échappe aux 5 minutes de stabilisation.

### 3.2. Renommage de l'alias

| Avant | Après |
|---|---|
| `Sommeil — Consolidation nuit evenementielle` | `Sommeil — Consolidation nuit` |

L'alias cesse de qualifier la nature du déclencheur (l'automation n'est plus exclusivement événementielle).
