Cette transition est une **phase d’industrialisation / durcissement** :
- **Gros câblage d’infrastructure** (énorme delta `configuration.yaml`) + **explosion d’ajouts** (858 fichiers ajoutés, 103 supprimés).

## 🧱 Domaine — Infrastructure / configuration / observabilité

### Points majeurs
- `configuration.yaml` : **delta énorme** → phase de câblage.
- Ajout d’un **binary_sensor MQTT** `binary_sensor.zigbee2mqtt_online` (supervision Zigbee2MQTT).
- Ajout explicite de `timer: !include_dir_merge_named timers/` (les timers deviennent un **pilier**).
- `recorder` / `logbook` : grosses listes d’exclusions (notamment capteurs météo dynamiques) + disparition de la section `history` (au moins ici : elle n’est plus dans ce fichier).

### Risques
- Une exclusion trop large en `logbook/recorder` peut “faire disparaître” des signaux utiles en diagnostic.
- Si `timer:` est inclus mais que certains timers n’existent pas / sont mal chargés, on introduit des erreurs transverses (beaucoup de logique s’appuie dessus).

## 🏠 Domaine — Modes (Maison / Babysitting / Vacances)

### Points majeurs
- **Babysitting** n’est plus un état “dans” `input_select.mode_maison` :
  - création/utilisation de `input_boolean.mode_babysitting` (mode autonome).
  - ajout d’un timer + logique de reset (`reset_mode_babysitting_auto`), et fin de mode plus cadrée (réinitialisations + relances).
- `Mode Maison - Retour à Normal` : devient un **gros reset orchestré**
  - réactive aussi la désinfection ECS,
  - relance la gestion chauffage via l’automation centrale (au lieu de pousser une consigne en direct),
  - réactive des helpers transverses (clim hiver, bouclage visiteur),
  - annule des timers gardiens ECS.
- `Mode Maison - Passage en Vacances` : durcissement net
  - désactive planifications ECS (ponctuel + désinfection),
  - désactive bouclage visiteur,
  - bascule chauffage en Reduced **via script unifié** *si ViCare dispo* (sinon notif d’échec),
  - sécurise ECS en abaissant à **10°C** *si ViCare dispo* (sinon notif d’échec),
  - annule timers ECS,
  - armement via script unifié.

### Risques
- Nouveau couplage implicite : “Vacances” repose fortement sur `binary_sensor.vicare_api_disponible`.
- La cohérence globale dépend davantage du bon fonctionnement des scripts unifiés (orchestrations plus denses).

## 🔐 Domaine — Sécurité / Alarme

### Points majeurs (signal fort)
- **Armement automatique** : durcissement clair
  - double déclenchement (événement + vérif périodique),
  - confirmation anti faux-positifs (>= 30 s),
  - suppression de la condition “Babysitting via mode_maison” (logique réorientée).
  - action déléguée à `script.alarme_armer` avec `origine`.
- **Correction post-démarrage** :
  - présence “confirmée” de façon robuste (cohérence de 2 capteurs + ancienneté > 30 s),
  - réactivation immédiate de l’automation d’armement (on évite un trou de sécurité).
- **Désarmement auto** :
  - description recentrée (retour présence),
  - condition “alarme = armed_away” supprimée → on délègue au script (plus robuste, mais exige un script bien défensif),
  - au démarrage : présence confirmée via “last_changed > 30 s”.
- **Délai d’entrée / désarmement** : bascule majeure
  - passage en **timer** `timer.delai_entree`,
  - durée pilotée par `input_number.alarme_delai_entree`,
  - logique “porte garage = délai doublé”,
  - annulation si désarmement / présence détectée,
  - mode test : déclenchement silencieux (sans sirène) mais traçable.
- **Mode visiteur** :
  - logique identique, mais notifications enrichies (résumé du statut sécurité rétabli, tags/group).

### Risques
- Beaucoup de comportements alarme dépendent maintenant :
  - de `timer.delai_entree`,
  - de `input_number.alarme_delai_entree`,
  - de scripts (armer/désarmer/clavier).
- Si un seul de ces piliers est absent/mal chargé : effet domino (armement auto, délais, notifications).

## 🔥 Domaine — Chauffage / ViCare

### Points majeurs
- `Gestion chauffage avec présence` (geolocalisation) : **refonte de logique**
  - bascule programme ViCare **COMFORT ↔ REDUCED** (au lieu de pousser une température),
  - ne fait rien si chauffage coupé,
  - contrôle anti-silence (toutes les 10 min + au démarrage HA),
  - **anti-rebond** via `timer.chauffage_geoloc_antirebond`,
  - notification uniquement si **changement effectif**,
  - dépendance à `active_vicare_program` (attribut du climate) pour détecter l’état courant.
- Courbe de chauffe : ajout d’une condition **chauffage actif** avant d’appliquer pente/parallèle au démarrage.

### Risques
- Si l’attribut `active_vicare_program` n’est pas présent / diffère des valeurs attendues, le système peut :
  - re-tenter des bascules inutiles,
  - ou se tromper de “prog_actuel”.
- Si `timer.chauffage_geoloc_antirebond` manque → risques de “yo-yo”/multi-actions.

## 🚿 Domaine — ECS (cycle, watchdog, modes)

### Points majeurs
- Script `Chauffage ECS - Cycle unifié` : durcissement massif
  - passage `mode: restart` → `mode: single` (choix de sûreté),
  - verrou explicite via `input_boolean.ecs_cycle_en_cours` :
    - reset automatique si verrou bloqué > 5 min,
    - stop si verrou récent (anti double lancement),
  - watchdog via `timer.ecs_cycle_watchdog`,
  - ajout du mode `vaisselle` (nouvelle consigne `input_number.ecs_consigne_vaisselle`),
  - logique de fin : abaissement confirmé à **10°C**, et notification critique si échec.
- En mode “Normal” / “Vacances” : annulation systématique des timers gardiens ECS lors des transitions.

### Risques
- Nouvelle discipline : si le verrou ou les timers ne sont pas correctement gérés, on peut créer :
  - “blocage ECS” (cycle non relançable),
  - ou “surveillance fantôme” (watchdog non annulé).
- L’apparition d’un mode `vaisselle` impose une vérification que tout l’écosystème (helpers + UI) connaît l’option.

## ❄️ Domaine — Climatisation

### Points majeurs
- Automations “chauffage clim hiver” :
  - déclencheur étendu à `input_boolean.mode_babysitting`,
  - cas explicite “fin babysitting + mode maison != Normal” => désactivation.
- Scripts `heat/cool/dry` : durcissement d’exécution
  - séquence plus robuste (power relais, set hvac_mode, wait mode appliqué, puis set temperature),
  - notifications plus structurées.

### Risques
- Dépendance accrue aux états réels (`wait_template`) : si l’unité ne confirme pas le mode, la séquence peut échouer (c’est souvent un *bon* échec, mais à connaître).

## 🌦️ Domaine — Météo / Aération / Couleurs

### Points majeurs
- **Split arborescence** Météo : `arsenal/` vs `imprimerie/`.
- Binary sensors d’aération (RDC / étage) : évolution de logique
  - remplacement du calcul basé sur `rain1h` par un **booléen** `input_boolean.pluie_en_cours` (changement de contrat de donnée : “mesure” → “état”).
- Nettoyage/rotation massive des templates “couleurs/météo” (beaucoup de moves).

### Risques
- Changement de source pluie :
  - si `input_boolean.pluie_en_cours` n’est pas fiable ou pas alimenté, la recommandation d’aération peut perdre sa pertinence (ou devenir trop permissive).

## ⚡ Domaine — Coupure secteur (mode panne)

### Points majeurs (signal)
- `activation_mode_panne` : abandon des actions directes → appel à `script.mode_panne_coupure_secteur` (centralisation).
- `desactivation_mode_panne` : **ne restaure plus le chauffage**

### Risques
- En sortie de panne, si la gestion chauffage (geoloc) est indisponible, le chauffage ne sera plus “réaligné” par cette automation (changement de responsabilité assumé, mais à valider).
