## 1 — Ouvertures : normalisation N1/N2 et abstraction capteurs

La couche Ouvertures est normalisée autour du préfixe `contact_*`, remplaçant les anciens capteurs `capteur_*`. 
Les automatisations critiques (chauffage, alarme, aération) consomment désormais cette couche abstraite.

### Nouveaux capteurs agrégés

```
12_template_sensors/ouvertures/capteurs_contact_base.yaml
12_template_sensors/ouvertures/capteurs_contact_agreges.yaml
12_template_sensors/ouvertures/fermeture_fenetres_stable.yaml
```

### Suppression

```
12_template_sensors/ouvertures/capteurs_contact.yaml
```

### Correction post-reboot des capteurs "avec délai"

Les capteurs `fenetre_*_ouverte_avec_delai` sont refactorisés pour éliminer les faux positifs après redémarrage HA. 
La nouvelle logique compare `timer.last_changed` à `last_open` : le capteur ne passe ON que si le timer a expiré **après** la dernière ouverture réelle.

Capteurs concernés : `fenetre_sejour_ouverte_avec_delai`, `fenetre_chambre_parents_ouverte_avec_delai`, `fenetre_ouverte_maison_avec_delai`.

## 2 — Aération : consolidation du pipeline M1–M6

### Déclenchement M1 sur fait métier

Le pipeline abandonne les triggers sur capteurs physiques individuels au profit d'un trigger unique sur `input_boolean.aeration_confirmee`. 
Les anciens triggers `ouverture_immediate`, `ouverture_differee` et `ouverture_grace` sont supprimés.

### Fermeture canonique M2

Le trigger de fermeture repose désormais sur `binary_sensor.fenetres_maison_fermees_stable`, 
garantissant une fermeture globale stabilisée avant exécution de M2.

### Qualification/disqualification dédiées

La logique de qualification de l'aération est extraite du pipeline :

```
11_automations/ouvertures/qualification_aeration.yaml
11_automations/ouvertures/disqualification_aeration.yaml
```

Nouveau helper de suspension et horodatage renommé :

```
05_input_booleans/aeration/suspension_aeration.yaml
07_input_datetimes/aeration/reprise_thermique/debut_aeration.yaml
```

### Nouveau module M6

```
10_scripts/aeration/m6_fermeture_post_reouverture.yaml
00_documentation_arsenal/contrats/aeration_blocage_chauffage/m6_refermeture/1_refermeture_apres_reouverture.md
```

M6 formalise la refermeture propre du blocage après une réouverture survenue pendant le blocage actif.

## 3 — Chauffage : rationalisation des triggers décisionnels

L'automation `decision_centrale_trigger.yaml` est simplifiée. 
Les déclenchements directs sur `aeration_episode_en_cours` et `chauffage_blocage_aeration` sont supprimés. 
Le chauffage se déclenche désormais uniquement sur `binary_sensor.fenetre_ouverte_maison_avec_delai` (transitions off→on et on→off explicites), 
alignant le domaine chauffage sur la couche Ouvertures normalisée.

## Synthèse

La v9.6 finalise la normalisation du domaine Ouvertures en introduisant 
une couche `contact_*` abstraite consommée par l'ensemble des automatisations, 
et corrige les faux positifs post-reboot des capteurs "avec délai". 
Le pipeline aération est consolidé autour d'un déclenchement sur fait métier et d'une fermeture canonique stabilisée, complété par M6. 
La décision chauffage et les automations d'alarme sont alignées sur cette nouvelle couche.
