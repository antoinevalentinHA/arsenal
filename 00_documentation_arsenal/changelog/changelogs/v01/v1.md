## Préambule de gouvernance (pourquoi c’est une transition d’ère)
Cette transition formalise un basculement net de gouvernance :
1) **Arborescence “gouvernée” par préfixes numériques** (01…13) + recâblage central des includes.
2) **Politique d’historique inversée** : on passe d’une logique “j’exclus du bruit” à une logique “je n’enregistre que ce que je décide” (whitelist).
3) **UI segmentée** : explosion de dashboards par domaine + ressources Lovelace relocalisées en `/local` (moins dépendant HACS).

Résultat : **plus gouvernable** (lisible, triable, auditabile) mais **plus fragile** si un seul pilier (dossier/include/ressource/timer) manque.

## 2) 🗃️ Chantier structurant — Historique (Recorder) & Journal (Logbook) en whitelist (signal critique)
### 2.1 Recorder : d’“exclude” à “include”
- Avant : exclusions par glob (Browser Mod + météo dynamique, etc.).
- Après : **enregistrement par liste explicite d’entités** + **auto_purge** activé, rétention courte (7 jours), commit_interval ajusté.

### 2.2 Logbook : même bascule (whitelist)
- Avant : exclusions par glob.
- Après : **include.entities** (journal “choisi” : navigation scripts, événements ciblés, etc.).

### 2.3 Risques (priorité haute)
- **Perte d’observabilité** : tout ce qui n’est pas explicitement listé ne laisse plus de traces (historique &/ou logbook).
- **Perte de profondeur temporelle** : auto purge 7 jours = impossible d’analyser une dérive sur plusieurs semaines/mois.
- **Effets de bord** : capteurs statistiques / dashboards basés sur historique peuvent devenir muets si leurs sources ne sont pas incluses.

## 3) 🖥️ Chantier structurant — Lovelace & Dashboards (signal + bruit)
### 3.1 Ressources Lovelace : relocalisation en `/local`
- Plusieurs cartes basculent de `/hacsfiles/...` vers `/local/...`
- Ajouts de nouvelles ressources (ex : card-mod, layout-card, stack-in-card, template-entity-row, auto-entities), tout en conservant Browser Mod via HACS.

✅ Signal : **diminution de dépendance à HACS** + verrouillage des versions via fichiers locaux.  
⚠️ Risque : si les fichiers ne sont pas réellement présents dans `/config/www` → UI cassée (cartes introuvables).

### 3.2 Dashboards : segmentation forte + “Navigation”
- Suppression de dashboards historiques (ex : “Réglages”, “Diagnostics” tels qu’ils étaient déclarés ici).
- Déclaration d’un **dashboard Arsenal visible sidebar** + création d’un dashboard “Navigation” + multiplication de dashboards par domaine (chauffage, clim, déshumidificateur, éclairage, ECS, prises, VMC, volets, santé, ouvertures, mouvements, modes, NAS, etc.).

✅ Signal : **architecture UI par domaines** (lisible et scalable).  
⚠️ Risque : fichiers dashboard manquants / liens de navigation invalides / redondances si certaines vues existent encore ailleurs.

### 3.3 `dashboards/arsenal.yaml` : devient un “tableau de bord maître”
Le dashboard Arsenal s’épaissit fortement et introduit des sections explicitement gouvernées (signal) :
- Modes de la maison
- Confort thermique
- Vaisselle & bouclage ECS
- Températures
- Ventilation & ouvertures
(etc.)

Bruit associé : commentaires très abondants + ajustements visuels. Le signal est la **structuration fonctionnelle**.

## 8) Risques — classement (du plus critique au plus bénin)
### 8.1 Critique
- Recorder/logbook en whitelist : risque de “silence” diagnostic + perte d’historique non prévu + purge courte.
- Relocalisation Lovelace `/local` : risque UI bloquante si fichiers non présents.

### 8.2 Élevé
- Recâblage includes vers dossiers 01…13 : risque de trous fonctionnels si un dossier est incomplet.
- Migration automations → 11_automations : risque de régression si certaines automations n’ont pas d’équivalent.

### 8.3 Moyen
- Activation pyscript : erreurs au boot / complexité runtime.
- trusted_proxies étendus : surface de confiance réseau plus large.
- Multiplication dashboards : risque de liens morts / redondance / maintenance UI.

### 8.4 Faible (bruit)
- Ajustements volumineux de `button_card_templates.yaml` : beaucoup de cosmétique ; le signal est la standardisation des templates utilisés par Arsenal.
