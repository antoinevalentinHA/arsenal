# ARSENAL — Contrat fonctionnel
## ECS — Automation `10250000000026` — Gel et validation des données de fin de cycle

---

## 1. Objet

Ce contrat définit le comportement opposable de l'automation :

```
10250000000026
```

Cette automation constitue le producteur canonique de fin de cycle ECS. Elle intervient à l'échéance de la fenêtre d'inertie post-cycle pour :

- consolider les données du cycle
- figer les métriques diagnostiques
- valider le résumé final
- émettre le signal canonique `ecs_fin_cycle_signal`

Elle matérialise le point de clôture exploitable d'un cycle ECS.

---

## 2. Rôle dans l'architecture

Cette automation :

- est déclenchée par l'expiration de `timer.fenetre_inertie_chauffe_ecs`
- lit les capteurs dynamiques nécessaires à la consolidation finale
- fige les données de cycle dans des helpers persistants
- valide le résumé du cycle à partir de la température max réelle
- produit le signal canonique `input_boolean.ecs_fin_cycle_signal`

Elle ne :

- ne décide pas de l'ouverture d'un cycle
- ne modifie aucune consigne chaudière
- ne pilote aucun équipement
- ne réalise pas l'auto-ajustement des offsets
- ne consomme pas elle-même le signal qu'elle produit

---

## 3. Trigger canonique

Trigger unique :

```
event_type: timer.finished
entity_id: timer.fenetre_inertie_chauffe_ecs
```

Le déclenchement est événementiel. Le gel n'est pas défini par un temps réel absolu à la milliseconde près, mais par la réception de l'événement Home Assistant `timer.finished` émis pour `timer.fenetre_inertie_chauffe_ecs`.

Toute autre source de déclenchement est hors contrat.

---

## 4. Séquence contractuelle

L'automation suit la séquence canonique suivante :

1. Forcer la mise à jour des capteurs dynamiques nécessaires à la consolidation
2. Garantir que leur mise à jour est terminée avant lecture des variables
3. Calculer :
   - la durée du cycle
   - la température max cycle pur
   - la température max réelle (cycle + inertie)
   - le résumé validé
4. Écrire les valeurs figées dans les helpers persistants
5. Mettre à jour le miroir du résumé validé
6. Émettre le signal canonique `input_boolean.ecs_fin_cycle_signal`

> Le signal ne doit jamais être émis avant que les écritures figées aient été réalisées.

---

## 5. Données lues

### 5.1 Capteurs dynamiques

- `sensor.ecs_temperature_max_cycle`
- `sensor.ecs_temperature_max_reelle_cycle`
- `sensor.ecs_dernier_cycle_resume`

### 5.2 Références de calcul

- `input_datetime.ecs_cycle_debut_timestamp`
- `input_datetime.ecs_tmax_timestamp`
- `sensor.ecs_consigne_chaudiere_securisee`

### 5.3 Référence temporelle de déclenchement

- `timer.fenetre_inertie_chauffe_ecs`

---

## 6. Données écrites

L'automation écrit exactement les entités suivantes :

| Entité | Rôle |
|---|---|
| `input_number.ecs_duree_dernier_cycle_figee` | Durée figée du cycle |
| `input_number.ecs_duree_chauffe_reel_backup` | Backup persistant de durée |
| `input_number.ecs_temperature_max_figee` | Température max cycle pur |
| `input_number.ecs_temperature_max_reelle_figee` | Température max réelle (cycle + inertie) |
| `input_text.ecs_resume_dernier_cycle_fige` | Résumé validé figé |
| `input_text.ecs_dernier_cycle_resume` | Miroir du résumé validé |
| `input_boolean.ecs_fin_cycle_signal` | Signal canonique de fin de cycle |

Aucune autre entité ne doit être modifiée par cette automation.

---

## 7. Calculs opposables

### 7.1 Durée du cycle

Définition actuelle :

```
duree = tmax_timestamp - cycle_debut_timestamp
```

en minutes, si les deux timestamps sont valides et ordonnés correctement. Sinon : `duree = 0`.

### 7.2 Température max cycle pur

```
input_number.ecs_temperature_max_figee ← sensor.ecs_temperature_max_cycle
```

Cette valeur représente le maximum atteint pendant le cycle actif uniquement.

### 7.3 Température max réelle

```
input_number.ecs_temperature_max_reelle_figee ← sensor.ecs_temperature_max_reelle_cycle
```

Cette valeur représente le maximum atteint sur l'ensemble du cycle actif et de la fenêtre d'inertie post-cycle. Elle constitue la référence thermique réelle post-inertie.

### 7.4 Validation du résumé

Le résumé passe de `|pending` à `|oui` si toutes les conditions suivantes sont réunies :

```
duree > 0
ET temp_max_reelle > 0
ET t0 est présent et convertible
ET (temp_max_reelle - t0) >= 0.5 °C
```

Sinon le résumé passe à `|non`.

La validation s'appuie sur la température max réelle figée, et non plus sur le seul maximum atteint pendant le cycle actif.

---

## 8. Production du signal canonique

Cette automation est le producteur canonique de `input_boolean.ecs_fin_cycle_signal`.

**Règles opposables :**

- le signal est émis uniquement après gel complet des données
- un cycle consolidé produit exactement une émission du signal canonique
- le signal représente la disponibilité des données figées post-inertie pour les consommateurs aval

Cette automation n'acquitte jamais le signal. L'acquittement appartient au ou aux consommateurs.

---

## 9. Invariants opposables

1. L'automation ne se déclenche qu'à l'échéance de `timer.fenetre_inertie_chauffe_ecs`.
2. Le gel final est toujours postérieur à la fenêtre d'inertie.
3. Les données figées sont écrites avant émission de `ecs_fin_cycle_signal`.
4. La validation du résumé repose sur `temp_max_reelle`, pas sur `temp_max` seul.
5. L'automation est un producteur de signal, pas un consommateur.
6. L'automation ne réalise aucun calcul d'auto-ajustement.
7. L'automation ne publie aucune commande chaudière.
8. L'automation ne lit ni n'interprète aucun ACK.

---

## 10. Limites assumées

### 10.1 Nature non bloquante de `homeassistant.update_entity`

`homeassistant.update_entity` est non bloquant. L'automation introduit un délai minimal de 1 seconde entre `update_entity` et l'évaluation des variables pour limiter le risque de relire un état précédent.

Si des incohérences persistent en production malgré ce délai, une mitigation complémentaire doit être documentée contractuellement.

### 10.2 Dépendance aux capteurs dynamiques

L'automation dépend de la justesse de `sensor.ecs_temperature_max_cycle`, `sensor.ecs_temperature_max_reelle_cycle` et `sensor.ecs_dernier_cycle_resume`. Une erreur de ces capteurs pollue les données figées aval.

### 10.3 Définition actuelle de la durée

La durée figée reste actuellement calculée du début de cycle à `tmax_timestamp`. Elle ne constitue pas encore une durée thermique "réelle post-inertie étendue" distincte.

## 12. Remarque d'architecture

Cette automation est une automation de consolidation terminale.

Elle ne décide pas du métier thermique en amont. Elle ne consomme pas le signal en aval. Elle transforme une fin de cycle physique et une fenêtre d'inertie en vérité système figée et exploitable.
