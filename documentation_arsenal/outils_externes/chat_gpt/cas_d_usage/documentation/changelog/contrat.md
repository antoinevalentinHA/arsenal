# 🧠 ARSENAL — ChatGPT
# Cas d’usage — Documentation / Changelog

## 🎯 Objet

Ce document définit les **règles strictes et la forme attendue**
lors de l’utilisation de ChatGPT pour la **rédaction de changelogs Arsenal**.

Il constitue un **contrat d’usage spécifique**, applicable uniquement
aux interactions dont l’objectif est la production ou l’enrichissement
d’un fichier de type *changelog*.

---

## 🧱 Périmètre

Ce cas d’usage couvre exclusivement :

- la rédaction de blocs de changelog,
- l’enrichissement d’un changelog existant,
- la reformulation contrôlée d’entrées déjà actées.

Il ne couvre pas :

- l’analyse de décisions,
- la justification des choix,
- la validation fonctionnelle,
- la proposition d’évolutions.

---

## 🧠 Principe fondamental

Un changelog Arsenal est :

- un **document factuel**,
- orienté **état du système**, jamais intention,
- décrivant **ce qui a changé**, jamais pourquoi cela a été imaginé.

Toute dérive interprétative invalide la production.

ChatGPT n’est **jamais auteur** d’un changelog Arsenal.
Il agit exclusivement comme **outil de consolidation et de mise en forme**
d’informations déjà validées par l’utilisateur.

---

## 🧱 Règles d’utilisation

### Règle 1 — Aucune anticipation

ChatGPT ne doit **jamais** :
- anticiper la stabilité d’une évolution,
- qualifier une fonctionnalité comme « validée », « fiable » ou « consolidée »,
- extrapoler un statut non explicitement fourni.

---

### Règle 2 — Aucun jugement de valeur

Le changelog ne contient :
- ni appréciation qualitative,
- ni projection,
- ni formulation subjective.

Les termes tels que « robuste », « optimisé », « fiable », « abouti »
sont **interdits**, sauf instruction explicite.

---

### Règle 3 — Fidélité stricte au vocabulaire Arsenal

ChatGPT doit :
- reprendre **strictement** le vocabulaire fourni,
- respecter les termes métiers Arsenal,
- ne jamais « améliorer » une formulation existante.

---

### Règle 4 — Neutralité temporelle

Un changelog décrit un **état constaté**, à un **instant donné**.
Toute projection future est interdite.

---

## 🧾 Forme attendue

Domaine / Sous-système — description synthétique :
• point factuel 1
• point factuel 2
• point factuel 3

---

## 🛑 Causes d’invalidation immédiate

La sortie est invalide si elle contient :
- anticipation ou projection,
- qualification non fournie,
- jugement de valeur,
- vocabulaire non Arsenal,
- reformulation implicite,
- texte hors bloc de changelog.

---

## 📌 Statut

Document contractuel.
Toute exception doit être explicitement demandée.
