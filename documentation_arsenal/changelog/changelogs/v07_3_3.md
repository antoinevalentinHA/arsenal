## 📌 Sécurité — Refactor mode panne secteur (chauffage / ECS)

### Problème adressé

* actions chauffage directes hors décision centrale
* latence non maîtrisée au retour secteur
* exception ECS existante non formalisée contractuellement

### Architecture mise en place

* Chauffage

  • introduction d’un override explicite via `input_boolean.mode_confort_chauffage`
  • ajout de ce boolean aux triggers de la décision centrale chauffage
  • suppression complète de tout pilotage direct de consigne hors décision centrale

* ECS

  • maintien volontaire d’un lancement direct et ponctuel de cycle ECS
  • exception explicitement assumée, documentée et bornée

### Automatisations concernées

* `1004000000001`

  • activation de `input_boolean.mode_confort_chauffage` à l’entrée en mode panne
  • délégation complète à la décision centrale pour le chauffage
  • appel du script ECS dédié
  • notifications système associées

* `1004000000002`

  • désactivation explicite de `input_boolean.mode_confort_chauffage` au retour secteur
  • réinitialisation ciblée de la consigne ECS (10 °C)
  • sortie partielle et contrôlée du mode panne

### Script `mode_panne_coupure_secteur`

* recentré exclusivement sur :
  • la gestion ponctuelle de l’ECS
  • la notification système
* aucune action chauffage
* documentation strictement alignée avec le comportement réel

## 📌 Chauffage — Décision centrale : bypass de temporisation pour override prioritaire

### Contexte

* un override explicite constitue une **intention critique**
* cette intention ne doit subir **aucune temporisation générique**

### Modification apportée

* automatisation `10240000000001`

  • override prioritaire actif
  – appel immédiat de la décision centrale
  • tous les autres triggers
  – temporisation de stabilisation conservée (2 s)

### Impact architectural

* respect strict du pipeline :
  événement → intention → décision → action
* suppression de toute latence non maîtrisée
* aucun impact hors contexte override

## 📌 Chauffage — Correction UI « attente » vs « repos nominal »

### Problème identifié

* assimilation erronée de `Neutre` à une attente thermique
* confusion entre confort satisfait et attente réelle

### Clarification actée

* `Neutre` = abstention décisionnelle
* `Neutre ≠ attente thermique`
* `Neutre + Confort` = repos nominal

### Correction UI

* distinction explicite :

  • repos nominal
  – confort atteint
  – brûleur inactif

  • attente thermique
  – confort autorisé
  – confort non atteint

* règles UI :

  • `Neutre + Confort` → Confort satisfait / repos
  • `Neutre + ≠ Confort` → Attente confort

* alignement strict avec la charte UI Arsenal

Aucune modification du moteur de décision.

## 📌 Chauffage — Standby vs blocages : alignement contrat ↔ moteur ↔ UI

### Problème adressé

* standby interprété comme blocage absolu côté UI
* ordre d’évaluation incorrect dans la raison canonique

### Cause racine

* `chauffage_non_autorise` évalué avant `standby_force`
* écrasement de la raison standby

### Corrections apportées

* Contrat clarifié

  • `standby_force` = attente thermique contrôlée
  • surveillance active
  • reprise automatique attendue

* Capteur canonique corrigé

  • réordonnancement des causes dans `sensor.chauffage_raison_calculee`
  • `standby_force` évalué avant tout blocage absolu

* UI de synthèse alignée

  • standby retiré des traitements blocage
  • mapping UI :
  – 🔒 Standby → Attente thermique (BLEU)
  – ⛔ Poêle / aération → Bloqué (ORANGE)
  – 🔥 Brûleur actif → Actif (ROUGE)

Le diagnostic markdown existant était déjà correct et inchangé.
