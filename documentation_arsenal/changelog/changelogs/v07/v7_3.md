## 📌 1) ECS — CONSOLIDATION FIN DE CYCLE (signal canonique + ACK)

### Problématique traitée
L’auto-ajustement des offsets ECS était déclenché par des écritures “figées” (helpers) → signaux implicites, risques de rejeu, ambiguïté “fin de cycle”.

### Signal canonique introduit
La fin de cycle ECS devient un événement **explicite**, **unique**, **consommable** :

- **Source documentaire de fin de cycle** : `input_number.ecs_duree_dernier_cycle_figee`
- **Signal canonique** : `input_boolean.ecs_fin_cycle_signal`
- **Règle de consommation** : ACK explicite (reset du signal).  
  Si échec → pas d’ACK → le signal reste ON (rejeu possible / investigation).

### Production / consommation (séparation stricte)
- **Production** (émission du signal) :
  - `10_automations/vicare/ecs/signal_fin_cycle.yaml`
  - ID mentionné : `10130000000099`
  - Déclenchement : écriture d’une **nouvelle** durée figée valide
  - Rôle : émettre `input_boolean.ecs_fin_cycle_signal` (et uniquement ça)

- **Consommation** (auto-ajustement) :
  - `10_automations/vicare/ecs/auto_ajustement_offset.yaml`
  - ID : `10130000000088`
  - Déclenchement exclusif : `input_boolean.ecs_fin_cycle_signal` → `on`
  - Action :
    - appel `pyscript.ecs_autocorrect_offsets`
    - **ACK** : `input_boolean.ecs_fin_cycle_signal` → `off`
  - Garde-fou : `input_boolean.ecs_autocorrect_active = on`
  - Règle : **si échec → pas d’ACK**

### Signal net
- Fin de cycle ECS : explicite, rejouable, robuste redémarrage/reload
- Synchronisation “événement + ACK” contractualisée

## 📌 2) CHAUFFAGE — VERROU DÉCISIONNEL EN PRÉSENCE (autorisation cible réintégrée)

### Problématique traitée
En présence, le système pouvait rester en **Neutre/Eco** malgré un besoin thermique réel : court-circuit logique et inhibition d’une chauffe légitime.

### Principe renforcé
- La **présence** n’est pas une autorité thermique : elle n’accorde que le **droit de décider**.
- L’autorité thermique métier est `sensor.chauffage_autorisation_cible`.

> En présence, si `sensor.chauffage_autorisation_cible = comfort`, le chauffage ne doit pas rester en Neutre.

### Correctif (Décision Centrale)
Fichier : `09_scripts/vicare/chauffage/decision_centrale.yaml`

- Présence :
  - le mode désiré est désormais dérivé de `sensor.chauffage_autorisation_cible`
  - la raison associée est désormais `sensor.chauffage_raison_calculee` (au lieu d’un libellé “presence_on”)

- Nettoyages structurants :
  - suppression de la branche `binary_sensor.pre_confort_actif`
  - simplification du cas “absence” côté raison (retour “absence”)

### Signal net
- Suppression d’un faux “repos thermique” possible en présence
- Verrou logique : la décision suit l’autorité thermique, pas un raccourci présence

## 📌 3) CHAUFFAGE — CAPTEURS DIAGNOSTIC ALIGNÉS (décision ↔ UI)

### Principe
`sensor.chauffage_mode_calcule` est repositionné explicitement comme **capteur DIAGNOSTIC** :
lisible, explicatif, **non consommable** comme autorité d’action.

### Alignement
Fichiers :
- `11_template_sensors/vicare/chauffage/diagnostic/mode.yaml`
- `11_template_sensors/vicare/chauffage/diagnostic/raison.yaml`

- Hiérarchie explicitée :
  - override utilisateur (`input_boolean.mode_confort_chauffage`)
  - blocages majeurs (système, aération, fenêtres, vacances, poêle, etc.)
  - présence : mapping depuis `sensor.chauffage_autorisation_cible` (Confort / Neutre / Eco)
  - absence : Eco

Objectif : suppression de toute divergence décision ↔ capteurs ↔ UI.
