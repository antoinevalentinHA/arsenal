## 🧭 Préambule de gouvernance — Transition d’ère

ARSENAL v7 acte une transition structurante : le système devient **gouverné** par une documentation canonique (`documentation_arsenal/`) et une UI refondue autour d’une **bibliothèque de templates segmentée par domaines**.

Pivot généralisé :
- **Décision = vérité métier explicite** (capteur / binaire d’autorisation)
- **Action = automation applicatrice** (convergence idempotente vers la décision)
- **UI = consommation** (aucune logique métier dispersée dans les dashboards)

## 🧱 Corrections structurelles

### Notifications — Synchronisation booléens d’aération (ID `1001000000006`)

**Problème :**
- `choose` imbriqués
- évaluation exclusive d’une seule zone
- seconde zone jamais synchronisée

**Correction :**
- suppression des `choose` internes
- remplacement par des blocs `if` indépendants
- évaluation systématique des deux zones

**Résultat :**
- synchronisation exhaustive et fiable
- idempotence conservée
- intention métier inchangée

## ⚡ Énergie — Verrouillage définitif des sources

### Règle ferme (irrévocable)
Le dashboard **Énergie Home Assistant** consomme **exclusivement** des capteurs énergie **strictement monotones, sécurisés et persistants** :
- proxys énergie Arsenal (`*_energy_proxy`)
- `device_class: energy`
- `state_class: total_increasing`
- protections : régressions, valeurs négatives, resets, états transitoires

### Interdits
- capteurs énergie Zigbee bruts
- `utility_meter`
- capteurs calculés non monotones
- toute entité susceptible de revenir à zéro

### Séparation stricte des rôles
- **Proxy énergie**
  - vérité long terme
  - source unique du dashboard Énergie
  - source des `utility_meter`
- **Utility_meter**
  - découpe temporelle uniquement
  - jamais exposé au dashboard Énergie
- **Capteurs Zigbee bruts**
  - sources techniques instables
  - jamais utilisés directement

### Conséquence opérationnelle
Toute modification de source Energy sur un appareil existant est **interdite**.
En cas de changement :
- suppression de l’appareil Énergie
- redémarrage Home Assistant
- recréation avec la source correcte

## 🧼 ECS — Mode Vaisselle : simplification architecturale

Suppression de l’automatisation :
- **« Mode Vaisselle – Activation globale des automatisations »** (ID `1013000000016`)

**Raisons :**
- pilotage dynamique d’automatisations = anti-pattern Arsenal
- l’état métier doit être déductible des entités exposées
- les automatisations évaluent déjà `mode_vaisselle = ON`

**Modèle retenu :**
- automatisations ECS toujours actives
- légitimité évaluée localement via conditions métier
- décision finale centralisée dans le script wrapper ECS

## 🖥️ UI — Consolidation architecturale globale (chantier clos)

### Principes UI verrouillés (sans exception)
- un dashboard est strictement déclaratif
- un dashboard ne décide rien
- un dashboard ne calcule rien
- aucun JS de décision
- logique UI externalisée en templates
- logique métier portée uniquement par :
  - capteurs / binary_sensors
  - scripts
  - automatisations

### Externalisation systématique par templates
- cartes button-card : factorisées, externalisées, organisées par rôle
  - état réel
  - décision / intention
  - diagnostic
  - action explicite
  - supervision
  - génériques réutilisables

### Dashboards emblématiques consolidés
- Arsenal
- Chauffage
- Climatisation
- Déshumidificateur
- Supervision NAS
- + revue transverse : aucun dashboard button-card exclu

**Résultat global :**
- UI homogène
- lisibilité immédiate des états et décisions
- dette UI historique supprimée
- ajout/modification de cartes sans effet transversal
