## 📌 Domaine — UI (Socles / KPI / Couleurs)

### Socle KPI — changement discret, impact massif (palette & fallback)
- Refonte de la logique de **palette** côté socle KPI (profil / fallback) :
  - Suppression du profil `meteo_legacy` en tant que fallback implicite.
  - **Fallback par défaut** : `palettes.arsenal` (au lieu de `palettes.meteo_legacy`).
- Ajustement de la valeur “yellow” dans la palette `arsenal` :
  - `yellow` passe à `rgba(255, 235, 59, 0.2)` (au lieu du jaune Arsenal canon historique `rgba(255, 193, 7, 0.25)` dans ce socle).
- Conséquence : **toutes** les cartes KPI utilisant le socle héritent d’un comportement plus contractuel (profil Arsenal par défaut) — sans “bruit visuel”, mais avec une portée transversale énorme.

### Capteurs “couleur_*” — observabilité intégrée (raison)
Ajout d’un attribut `reason` sur les capteurs couleur afin de distinguer explicitement :
- `unavailable` (source absente/indispo ou non numérique exploitable)
- `invalid_thresholds` (seuils incohérents : `bas == 0` / `haut == 0` / `bas >= haut`)
- `ok` (calcul nominal)

Concerne explicitement :
- `11_template_sensors/couleurs/temperature.yaml` → `sensor.couleur_temperature_*`
- `11_template_sensors/couleurs/humidex.yaml` → `sensor.couleur_humidex_*`
- `11_template_sensors/couleurs/humidite_relative.yaml` → `sensor.couleur_humidite_relative_*`

> Effet système : le rendu UI peut rester strictement identique, mais le diagnostic devient **objectivable** (indispo vs seuils invalides vs OK), ce qui est critique pour l’anti-“unknown” et les enquêtes de dérives.

## 📌 Domaine — Aération (UI & pipeline)

### Dashboard Aération — Synthèse : simplification volontaire de la priorité
- `button_card_templates/dashboards/aeration/diagnostic_synthese.yaml`
  - Priorité passe de `blocage > episode > deltaT > nominal` à `blocage > episode > nominal`.
  - Retrait complet de la branche “chute thermique résiduelle” basée sur `input_number.delta_t_max_decisionnel_aeration`.
  - Harmonisation couleurs :
    - indispo → `rgba(224, 224, 224, 0.2)` (au lieu de `rgba(158, 158, 158, 0.1)` dans ce template)
    - épisode → bleu info Arsenal `rgba(144, 202, 249, 0.25)` (au lieu d’un bleu non canon `rgba(33, 150, 243, 0.2)` ici)

### Dashboard Aération — ΔT : rendu “actif uniquement”, et attente d’analyse explicitée
- `button_card_templates/dashboards/aeration/diagnostic_delta_t.yaml`
  - Ajout de triggers_update structurants : `input_boolean.aeration_episode_en_cours`, `input_boolean.chauffage_blocage_aeration`, `timer.aeration_analyse_delta_t` (en plus de `input_number.aeration_delta_t_utilise`).
  - Le label devient **contextuel** :
    - si ni épisode ni blocage → affichage neutre `"—"` (donc carte “muette” hors pipeline)
    - si `timer.aeration_analyse_delta_t` actif → `"Analyse ΔT en attente"`
    - sinon : interprétation de `input_number.aeration_delta_t_utilise` (faible/modérée/importante)

> Conséquence : la carte ΔT ne “pollue” plus l’écran hors épisode/blocage, et rend visible l’état *analyse en attente*.
