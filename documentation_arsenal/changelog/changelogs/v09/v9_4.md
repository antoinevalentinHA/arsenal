## 2 — Résilience des intégrations

Introduction d'un **moteur générique de récupération des intégrations** avec compteur de tentatives, timer de backoff et reset automatique au retour des données. 
Les automations précédentes basées sur `time_pattern` sont remplacées par un modèle événementiel avec backoff réel.

### Script générique

```
09_scripts/system/resilience_integration_recover.yaml
```

### Timers backoff (un par intégration)

```
08_timers/reload_integrations/*
```

### Automations de reload

```
10_automations/system/reload_integrations/*
```

### Refactorisation Zigbee2MQTT

L'automation reload Zigbee est entièrement réécrite : backoff progressif réel (10 → 60 min, cap 60), 
vérification post-restart par `wait_template`, plafond à 5 tentatives consécutives, 
reset automatique dès 2 minutes de fonctionnement stable du bridge.

## Synthèse

La v9.4 marque une évolution structurante d'Arsenal :

• refactorisation du moteur d'aération en pipeline modulaire
• introduction d'un moteur de résilience générique pour les intégrations
• consolidation de l'observabilité système
• introduction de la stratégie volets pluie

Elle étend également le système de capteurs couleurs aux domaines
santé et ambiance et enrichit les diagnostics UI.
