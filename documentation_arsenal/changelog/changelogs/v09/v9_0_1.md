## 📌 Signal global (ce qui change vraiment)

- **Chauffage : souveraineté opérateur institutionnalisée**
  - `input_boolean.mode_confort_chauffage` devient un **override souverain** (niveau 0), prioritaire sur le régime automatique.
  - Conséquence : une commande humaine explicite ne doit plus être contrariée par des abstentions / garde-fous logiciels (ex. anti-rebond, programme inconnu), hors interdits/sécurités absolues.

- **UI Chauffage : bascule vers “observable only”**
  - Notification persistante Chauffage **réduite à un seul cas** : **Confort uniquement**.
  - Source UI **exclusive** : `sensor.programme_chauffage` (et non une mémoire souveraine).

- **ECS : passage au niveau “constitutionnel”**
  - Décomposition du contrat ECS monolithique en **corpus normatif spécialisé**.
  - L’ancien contrat est **déclassé** et reclassé en **référence historique**.

## 📌 Chauffage — Décision centrale (gouvernance & logique)

### Override opérateur (Niveau 0)
- `mode_confort_chauffage = on` :
  - impose `comfort` **avant** toute hiérarchie automatique,
  - **bypass contrôlé** de garde-fous d’abstention (notamment anti-rebond, programme inconnu),
  - raison métier associée : `confort_force` (statut diagnostique de premier rang).

### Hiérarchie automatique (hors override)
- La hiérarchie “blocages / contextes / présence / absence” est conservée,
  mais explicitement requalifiée comme **régime automatique** (subordonné à l’override).

**Point structurant :**
- L’override n’est plus une “autorisation forcée” : c’est une **commande souveraine** traitée distinctement dans la doctrine.

## 📌 Chauffage — Capteurs diagnostics (miroir contractuel)

- `sensor.chauffage_mode_calcule` :
  - intègre explicitement le niveau 0 (override),
  - formalise des cas de protection thermique (ex. absence + inhibition géofencing) conformément à la table de décision.

- `sensor.chauffage_raison_calculee` :
  - ajoute `confort_force` en tête de hiérarchie (niveau 0),
  - réordonne les causes : override → blocages hiérarchiques → attente autorisation → standby → opportunités → absence.

**Signal net :**
- Les diagnostics deviennent un **outil d’audit** aligné sur la table de décision canonique (et non un résumé vague d’états).

## 📌 UI — Notifications persistantes (Chauffage)

### Changement majeur de doctrine
- Avant : notifications persistantes (Confort/Réduit) reconstruites depuis une mémoire souveraine.
- Maintenant :
  - **une seule** notification persistante : **Confort**
  - **source exclusive** : `sensor.programme_chauffage`
  - déclenchement :
    - entrée en Confort / sortie du Confort
    - reconstruction post-démarrage via `input_boolean.systeme_stable → on`
  - disqualification explicite dès sortie de Confort
  - **interdiction** : notification persistante pour le mode Réduit

**Signal net :**
- La notification persistante devient une **projection UI observable**, découplée de toute “mémoire souveraine” de décision/exécution.
