## ❄️ 2) Climatisation — Refactoring décisionnel (pipeline canonique)

### Autorité de décision
- Création de l’autorité décisionnelle pure :
  - `sensor.clim_target_mode` (cool / dry / heat / off)
- Décision dérivée exclusivement des besoins + autorisations climatiques
- Suppression de toute logique métier hors pipeline canonique

### Exécution
- Mise en place du script idempotent unique :
  - `script.clim_execution`
- Application du mode cible uniquement
- Aucune commande envoyée si l’état réel est déjà conforme

### Transit décisionnel & cohérence
- Création de l’automation de transit décisionnel :
  - `automation.10030000000105`
- Ajout du diagnostic de cohérence :
  - `binary_sensor.clim_incoherence_decision_reel`
  - détection divergences persistantes décision ↔ réel (anti-bruit ≥ 60 s)
- Création de l’automation watchdog :
  - `automation.10030000000106`
  - réalignement idempotent automatique via pipeline officiel

### Autorisations métier
- Intégration de l’absence prolongée comme contrainte exclusivement sur l’autorisation COOL
- DRY et HEAT inchangés (règles existantes)

### Nettoyage structurel
- Suppression du script monolithique historique :
  - `script.clim_decision_centrale`
- Suppression de l’automation :
  - "Clim – Analyse complète et décision" (ancien ID : 1003000000100)
- Suppression des automatisations d’action directe devenues redondantes / incorrectes
- Suppression du fichier :
  - `10_automations/climatisation/cool/absence_longue.yaml`
  (après intégration complète dans l’autorisation COOL)

### Guard (clarification de rôle)
- Le GUARD est explicitement “indépendant de la décision centrale” :
  - il impose des invariants, corrige l’exécution réelle si besoin
  - il ne re-décide jamais et ne remplace pas l’autorité de décision
