## 📌 Lecture rapide (ce qui compte vraiment)

### ✅ SIGNAL (évolutions structurantes)
1) **Pivot “BOOT” → “SYSTÈME STABLE”**  
   Plusieurs automatisations critiques ne se déclenchent plus sur `homeassistant: start` mais sur **`input_boolean.systeme_stable` → on**, avec en plus la suppression de délais “à l’aveugle” (ex: suppression d’un `delay 2 min` côté correction démarrage courbe de chauffe).  
   ➜ C’est un durcissement net : le “boot” brut est jugé trop sale, on attend un état contractuel “stable”. :contentReference[oaicite:0]{index=0}:contentReference[oaicite:1]{index=1}

2) **Refactor / extension du domaine ALARME (helpers + scripts + timers)**
   Apparition d’un lot conséquent d’artefacts “decision/intention/état cible/timeout”, avec timer dédié et scripts centraux.  
   ➜ Ça ressemble à une industrialisation “décision centrale” pour l’alarme (intention → décision → exécution → timeout). :contentReference[oaicite:2]{index=2}

3) **Re-structuration des helpers (arborescence plus lisible)**
   On découpe des fichiers “fourre-tout” en sous-dossiers à thème (ex: `eclairage/duree_absence/*`, `meteo/zones_vertes_dashboard/*`, `system/bssid`, `voiture/autonomie`, `climatisation/sauvegarde_etat_avant_dry`, imprimerie atelier vs stock…).  
   ➜ Signal clair : normalisation documentaire + évolutivité. :contentReference[oaicite:3]{index=3}

4) **Durcissements chauffage / ViCare (déclencheurs + simplifications)**
   Le “trigger” décision centrale chauffage passe lui aussi sur `systeme_stable`, et on retire des morceaux de logique “pré-check boot” (wait_template + branche boot + délai 30s) au profit d’une stabilisation courte (2s) et d’un déclenchement contractuel.  
   ➜ Attention : c’est un changement de temporalité + de garde-fous. :contentReference[oaicite:4]{index=4}

## 🧩 Évolutions “comportement” (extraits significatifs)

### 1) Boot sécurisé via `input_boolean.systeme_stable`
Cas observés dans le digest :
- chauffage (courbe de chauffe) : trigger boot → stable + suppression délai fixe (2 min) :contentReference[oaicite:8]{index=8}
- chauffage (décision centrale trigger) : boot → stable + retrait garde “wait_template boot” + retrait délai 30s “boot” :contentReference[oaicite:9]{index=9}
- système (BSSID Wi-Fi) : ajout d’un trigger “stabilisation HA” sur `systeme_stable` :contentReference[oaicite:10]{index=10}
- visiteur correction au démarrage : boot → stable :contentReference[oaicite:11]{index=11}
- clim “verif_demarrage” : boot → stable :contentReference[oaicite:12]{index=12}

➡️ Interprétation opérationnelle (sans blabla) : **`systeme_stable` devient un “sémaphore” de gouvernance.**  
Si ce booléen est mal piloté (jamais ON, ON trop tôt, ON instable), **tu perds des initialisations “après reboot”** sur plusieurs domaines (chauffage, clim, alarme/visiteur, BSSID…).
