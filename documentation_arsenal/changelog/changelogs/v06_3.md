🧠 ARSENAL HA — v6.3 STABLE
==============================

Stabilisation post-ingénierie, formalisation explicite d’un pilier Chauffage (autorisation), durcissement Poêle, et sécurisation fondatrice de l’architecture Energy.

Cette version acte la bascule : Arsenal cesse d’être un chantier actif, et devient un système durable, autonome, vérifiable.

--------------------------------------------------
🧱 1. CLÔTURE — STABILISATION RECORDER & OBSERVABILITÉ
--------------------------------------------------

Après une période d’évolutions structurelles (recorder, entités, pipelines décisionnels, sémantique métier), un repack manuel Recorder a été réalisé.

Cet acte marque explicitement :
- la fin de la phase d’ingénierie lourde côté stockage/historique
- la validation d’une base saine et cohérente
- le passage en régime stable et durable

À partir de cette version :
- aucune maintenance lourde n’est attendue
- aucun repack n’est planifié
- le système est conçu pour fonctionner en autonomie

--------------------------------------------------
🔥 2. CHAUFFAGE — AUTORISATION THERMIQUE EXTERNALISÉE & APPLICATEUR IDÉMPOTENT
--------------------------------------------------

Objectif : sortir définitivement la logique d’autorisation “au fil des automatisations” et la rendre :
- unique,
- lisible,
- observable,
- non dupliquée,
- strictement séparée entre **décision** et **application**.

### 2.1 Décision = vérité métier centralisée (capteur)
Introduction d’une vérité métier dédiée à l’autorisation Chauffage, calculée côté template sensor :
- `sensor.chauffage_autorisation_cible`

Cette vérité devient la source unique pour la cible ternaire :
- `comfort`
- `neutre`
- `reduced`

La logique d’autorisation n’est plus “réécrite” dans l’automatisation : elle est **portée** par le capteur.

### 2.2 Application = automatisation applicatrice (sans logique)
Refonte de l’automatisation Chauffage :
- l’automatisation ne calcule plus
- elle **applique** uniquement la cible issue de `sensor.chauffage_autorisation_cible`
- elle pilote uniquement le verrou logique :
  - `input_boolean.chauffage_standby_force`
- elle se synchronise aussi sur :
  - `input_boolean.systeme_stable`

Résultat :
- **idempotence** (pas d’actions répétées inutiles)
- **zéro duplication métier**
- **auditabilité** : une lecture du capteur suffit à expliquer le choix

### 2.3 Sémantique diagnostic : “attente” explicite
Durcissement de la lisibilité backend/UI :
- remplacement d’un état “négatif” / ambigu par un état métier lisible :
  - `chauffage_non_autorise` → `chauffage_en_attente`

Le diagnostic devient plus exact : ce n’est pas “interdit”, c’est “pas encore requis / pas activable selon la vérité métier”.

### 2.4 Alignements UI Chauffage
Évolutions côté dashboards (diagnostics + réglages) :
- exposition explicite de `sensor.chauffage_autorisation_cible`
- mise à jour des libellés et sections pour refléter les nouveaux états (attente, blocages, verrou)

🪵 3. POÊLE — BLOCAGE À DURÉE CONFIGURABLE (FIN DU “HARDCODE”)
--------------------------------------------------

Objectif : rendre le blocage Poêle réglable, stable et visible, sans logique cachée.

### 3.1 Nouveau réglage utilisateur
Introduction d’un input_number dédié :
- `input_number.poele_duree_blocage_chauffage_minutes`

### 3.2 Application concrète sur le timer
Lors de l’activation du blocage Poêle, la durée du timer est désormais calculée depuis le réglage :
- `timer.poele_blocage_chauffage` reçoit une `duration` issue de `input_number.poele_duree_blocage_chauffage_minutes`

### 3.3 Automation dédiée d’application
Ajout d’une automatisation dédiée :
- application / prise en compte du réglage de durée (logique “applicatrice”)

### 3.4 Alignements UI Poêle
Mise à jour des dashboards :
- exposition du timer, de l’état de blocage, et de la durée configurée
- disparition des traces / états historiques devenus non pertinents

--------------------------------------------------
🧱 4. ÉNERGIE — SÉCURISATION PAR PROXYS MONOTONES (ZIGBEE)
--------------------------------------------------

Objectif : éliminer une classe d’instabilité du dashboard Énergie HA (valeurs négatives / reset de prise / incohérences), et rendre la chaîne Energy “robuste par construction”.

### 4.1 Problème identifié (signal)
Les capteurs énergie Zigbee “bruts” peuvent :
- repartir à zéro après coupure
- dériver / corriger
- provoquer des deltas négatifs
→ ce qui détruit la cohérence des statistiques Energy HA.

### 4.2 Décision d’architecture (fondatrice)
Interdiction d’utiliser un capteur énergie Zigbee brut comme source directe Energy.

Toute source `sensor.*_energy` issue d’une prise Zigbee DOIT transiter par un proxy monotone persistant :
- `sensor.deshumidificateur_energy_proxy`
- `sensor.refrigerateur_energy_proxy`
- `sensor.lave_vaisselle_energy_proxy`

### 4.3 Migration utility_meter
Bascule des `utility_meter` vers les proxys :
- remplacement des `sensor.prise_*_energy` par `sensor.*_energy_proxy`

### 4.4 Persistance & observabilité
- intégration explicite des proxys dans `recorder`
- ajout / alignement dans `logbook`
- objectif : survivre à un redémarrage / une coupure sans re-casser l’historique

### 4.5 Assainissement Energy (bruit maîtrisé, mais nécessaire)
Nettoyage des statistiques Energy contaminées par l’ancien modèle (capteurs bruts).
(L’action est “opérationnelle” côté HA, pas un changement de logique Arsenal.)
