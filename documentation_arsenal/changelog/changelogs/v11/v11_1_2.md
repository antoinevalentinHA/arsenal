# 1. Climatisation — Correction critique + fiabilisation des seuils

## 1.1 Correction critique — `binary_sensor.clim_seuil_extinction_heat_atteint`

### Problème avant v11.1.2

Le capteur `binary_sensor.clim_seuil_extinction_heat_atteint` était structurellement incorrect.

Au lieu de retourner une condition booléenne, il retournait une valeur numérique calculée :

```jinja
{% set consigne = states('sensor.temperature_consigne_appliquee_locale') | float(20) %}
{% set offset_off = states('input_number.clim_offset_off') | float(0.5) %}
{{ (consigne - offset_off) | round(1) }}
```

Conséquences :

- ce `binary_sensor` ne produisait pas une vérité booléenne
- aucune comparaison n’était effectuée
- son comportement dépendait de coercions implicites Home Assistant
- la logique d’extinction HEAT pouvait donc être fausse tout en restant silencieuse

### Correction apportée

Le capteur a été réécrit pour :

- produire une condition booléenne explicite
- comparer la température minimale réelle au seuil OFF canonique
- retourner `false` si une entrée est `unknown`, `unavailable` ou vide

Principe corrigé :

```jinja
{% set temp_min = states('sensor.temperature_min_chambres') %}
{% set seuil_off = states('sensor.seuil_extinction_chauffage_clim') %}

{% if temp_min in ['unknown', 'unavailable', ''] or
      seuil_off in ['unknown', 'unavailable', ''] %}
  false
{% else %}
  {{ temp_min | float >= seuil_off | float }}
{% endif %}
```

### Impact

- rétablissement du fonctionnement réel du seuil d’extinction HEAT
- suppression d’un comportement structurellement faux
- fiabilisation de toute la chaîne ON/OFF du mode HEAT

👉 C’est la correction la plus importante de la release côté climatisation.

## 1.2 Fiabilisation des autres capteurs de seuils

Les capteurs suivants ont été harmonisés :

- `binary_sensor.clim_seuil_allumage_cool_atteint`
- `binary_sensor.clim_seuil_extinction_cool_atteint`
- `binary_sensor.chambre_max_humidex_au_dessus_seuil`
- `binary_sensor.chambre_max_humidex_en_dessous_seuil_off`
- `binary_sensor.clim_seuil_allumage_heat_atteint`

### Problème

Les versions précédentes convertissaient directement les états avec des valeurs par défaut du type :

```jinja
| float(-99)
| float(999)
```

Cela pouvait transformer :

- `unknown`
- `unavailable`
- `''`

en valeurs numériques artificielles, donc en états logiques faux mais acceptés.

### Correction

Tous ces capteurs utilisent désormais un guard explicite :

- si une entrée n’est pas exploitable → `false`
- sinon comparaison réelle

### Impact

- fin des faux positifs silencieux
- suppression des activations indues sur données invalides
- capteurs désormais déterministes et contractuellement propres

# 2. Alarme — Correction structurelle du blocage d’armement

## 2.1 Problème

La temporalité du blocage d’armement automatique n’était pas suffisamment canonique.

Une partie du comportement reposait encore sur une logique locale de script, avec une temporalité embarquée.

Dans Arsenal, c’est une mauvaise idée pour trois raisons :

- non observable
- non restaurable proprement après redémarrage
- propice aux incohérences blocage ↔ temporalité

---

## 2.2 Correction

La temporalité a été externalisée vers un mécanisme dédié :

```yaml
timer.blocage_armement_auto
```

Le script de désarmement ne porte plus lui-même la durée du blocage.  
Il active un état, et la levée relève désormais du mécanisme canonique.

En parallèle :

- formalisation du contrat de blocage d’armement
- ajout d’un watchdog dédié
- clarification de la séparation entre :
  - logique nominale
  - cohérence structurelle
  - correction d’anomalie

---

## 2.3 Invariant restauré

L’invariant attendu devient explicite :

```text
input_boolean.blocage_armement_auto == on  ↔  timer.blocage_armement_auto actif
timer.blocage_armement_auto == idle        ↔  blocage levé
```

---

## 2.4 Impact

- comportement boot-safe
- temporalité observable
- suppression du risque de blocage orphelin
- correction minimale et déterministe en cas d’écart

👉 Ce n’est pas une amélioration de confort.  
C’est une réparation d’architecture.

# 3. Boiler — Alignement du contrat sur la réalité du bridge

## 3.1 Problème

La documentation du bus MQTT chaudière décrivait encore un modèle trop hérité :

- programme
- confort
- réduit

alors que l’implémentation réelle du boiler bridge expose une consigne chauffage active unique.

Le contrat racontait donc encore une histoire partiellement fausse.

---

## 3.2 Correction

Le namespace canonique et la politique de payload ont été réalignés.

### Évolutions principales

- suppression des commandes obsolètes liées à l’ancien modèle chauffage
- passage à un modèle unifié centré sur :
  - `boiler/command/heating/set_temperature`
  - `boiler/ack/heating/set_temperature`
  - `boiler/telemetry/heating/setpoint`
- ajout de la télémétrie :
  - `boiler/telemetry/burner/modulation`
- mise à jour des exemples de payload et d’erreur

---

## 3.3 Principe clarifié

Le contrat réaffirme désormais explicitement :

- la chaudière expose la réalité technique
- Arsenal porte seul la sémantique métier chauffage

Autrement dit :

- plus de pseudo-sémantique “confort / réduit / programme” côté bridge
- plus d’ambiguïté sur la frontière d’autorité

---

## 3.4 Impact

- fin du décalage contrat ↔ implémentation
- simplification conceptuelle
- meilleure lisibilité de la frontière chaudière / Arsenal

👉 Là encore, la version enlève une fausse couche de sens.
