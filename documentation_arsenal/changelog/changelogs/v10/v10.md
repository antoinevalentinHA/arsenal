## 🧭 PRÉAMBULE DE GOUVERNANCE

La v10 clôt le cycle v9.x sur le pipeline Ouvertures / Aération.

Elle introduit une formalisation explicite de la fin de grâce,
une redéfinition de la qualification d’ouverture,
et une consolidation monotone complète des échéances thermiques.

Cette version élimine définitivement :

- toute interprétation métier basée sur `timer = idle`
- toute ambiguïté entre `timer.finished` et `timer.cancel`
- toute levée de blocage non hiérarchisée
- toute survivance d’état zombie post-reboot

Les principes introduits sont irréversibles.

---

## 📌 1. Fin de grâce explicitée (writer unique)

### Ajouts structurels

- `input_boolean.sejour_grace_echue`
- `input_boolean.chambre_parents_grace_echue`

### Modification clé

Les templates :

- `fenetre_sejour_ouverte_avec_delai`
- `fenetre_chambre_parents_ouverte_avec_delai`

ne dépendent plus de `timer = idle`.

La condition devient :

'any_open AND grace_echue'


### Effet technique

- `timer.cancel` ne peut plus produire de faux ON
- aucun flash de confirmation possible
- causalité stricte : seul `timer.finished` active le booléen

Principe introduit :

> Un timer n’est plus interprété.  
> Il produit un marqueur métier explicite.

---

## 📌 2. Redéfinition de la qualification d’aération

### Modification majeure

`10_automations/ouvertures/qualification_aeration.yaml`

Remplacement :

- déclenchement direct `timer.finished`
→ par
- écoute des booléens `*_grace_echue`

Ajout :

- `binary_sensor.ouverture_qualifiee_maison`

### Nouvelle logique

Cas sans délai :
- wait court (3s)
- confirmation uniquement si `ouverture_qualifiee_maison = on`

Cas avec délai :
- confirmation seulement si fenêtre encore ouverte
- grâce explicitement échue

Effet :

- plus de confirmation tardive
- plus de confirmation après fermeture
- plus de dépendance à `contact_fenetres_maison`

## 📌 3. Consolidation monotone M2 / M3 / M6

### Scripts impactés

- `m2_fin_episode.yaml`
- `m3_1_prolonger_blocage.yaml`
- `m6_fermeture_post_reouverture.yaml`

### Corrections techniques majeures

- suppression des `.replace(tzinfo=None)`
- usage systématique de `as_timestamp`
- calcul en secondes strictement borné par `max()`
- formatage datetime via `as_local(...).strftime(...)`

### M6 refondu

- calcul hybride :
  - analyse = délai stabilisation
  - blocage = max(blocage_min, blocage_restant)
- hiérarchie garantie :
  analyse < blocage
- aucun raccourcissement autorisé

Principe introduit :

> Toute échéance thermique est monotone et hiérarchisée.
