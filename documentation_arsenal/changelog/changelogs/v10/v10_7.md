# Changelog Arsenal v10.7

Statut : PUBLIÉ  
Nature : Refactorisation architecturale / Observabilité / Normalisation des événements  
Portée : Mouvements, automatisations d’éclairage, système d’alarme, diagnostics d’ouvertures

---

# 1 — Vue d’ensemble

La version **v10.7** poursuit l’évolution architecturale d’Arsenal vers un modèle **événementiel par zones**.

Les capteurs matériels de mouvement ne sont plus utilisés directement par les automatisations.  
Ils sont désormais encapsulés dans des **capteurs agrégés de zone**.

Cette refactorisation produit plusieurs effets structurants :

- réduction du nombre de triggers physiques
- disparition des dépendances aux capteurs individuels
- meilleure robustesse face aux défaillances Zigbee
- lecture métier par **zones logiques**

En parallèle, cette version introduit un **sous-système de diagnostic de redondance des contacts** afin d’améliorer l’observabilité des ouvrants.

---

# 2 — Capteurs de mouvements agrégés (nouvelle couche N2)

## Introduction

Un nouveau socle de capteurs agrégés est introduit pour représenter l’activité des zones.

Entités principales :

```
binary_sensor.mouvement_sejour
binary_sensor.mouvement_entree
binary_sensor.mouvement_garage
```

Ces capteurs constituent désormais la **source événementielle officielle** pour :

- éclairage automatique
- détection d’intrusion
- logique d’activité domestique

## Implémentation

Nouveaux fichiers :

```
11_template_sensors/mouvements/capteurs_base.yaml
11_template_sensors/mouvements/capteurs_agreges.yaml
11_template_sensors/mouvements/global.yaml
```

Principe :

- les capteurs matériels (`capteur_mouvements_*`) restent en **N1**
- les capteurs `binary_sensor.mouvement_*` exposent une **activité de zone**
- les automatisations consomment exclusivement la couche N2

Objectif :

> Décorréler la logique métier de la topologie matérielle.

# 3 — Migration des automatisations vers les capteurs de zone

Plusieurs automatisations ont été refactorisées pour utiliser les capteurs `binary_sensor.mouvement_*`.

## Éclairage garage

Automatisations modifiées :

```
10_automations/eclairage/garage/allumage_automatique.yaml
10_automations/eclairage/garage/extinction_automatique.yaml
```

Ancien modèle :

```
binary_sensor.capteur_mouvements_garage_1_occupancy
binary_sensor.capteur_mouvements_garage_2_occupancy
binary_sensor.capteur_mouvements_garage_3_occupancy
```

Nouveau modèle :

```
binary_sensor.mouvement_garage
```

Effets :

- simplification des triggers
- suppression des vérifications "tous les capteurs = off"
- réduction du risque de blocage par capteur isolé

Bénéfices :

# 4 — Refactorisation de l’automatisation d’intrusion

L’automatisation d’alarme basée sur les mouvements a été réécrite.

Fichier :

```
10_automations/alarme/intrusion/mouvement.yaml
```

## Nouveaux triggers

```
binary_sensor.mouvement_sejour
binary_sensor.mouvement_entree
binary_sensor.mouvement_garage
```

## Conditions de déclenchement

Le déclenchement est inhibé lorsque :

```
timer.delai_entree = active
binary_sensor.roborock_q7_max_nettoyage = on
```

Le déclenchement dépend également du mode test :

```
input_boolean.mode_test_alarme
```

## Notifications

Les notifications identifient désormais explicitement la **zone d’origine du mouvement**.

Exemple :

```
Mouvement detecte a HH:MM dans la zone sejour
```

Objectif :

- réduire les déclenchements multiples
- améliorer la lisibilité des alertes
- aligner la sécurité sur la logique de zones

# 8 — Correctif automatisation ScanWatch

Correction d’une entité incorrecte dans l’automatisation de batterie.

Fichier :

```
10_automations/sante/notification_batterie_montre.yaml
```

Ancienne entité :

```
sensor.scanwatch_2_battery_local
```

Entité réelle :

```
sensor.scanwatch_batterie_local
```

Conséquence :

L’automatisation ne pouvait jamais se déclencher car l’entité surveillée n’existait pas.

Ce correctif rappelle une règle d’exploitation importante :

> Toujours vérifier l’existence réelle des entités dans les outils développeur avant toute analyse du comportement d’une automatisation.

# 10 — Synthèse

La version **v10.7** marque une étape importante dans la stabilisation de l’architecture événementielle d’Arsenal.

Principales évolutions :

- introduction des **capteurs de mouvements agrégés par zone**
- migration des automatisations vers cette couche
- refactorisation de la détection d’intrusion
- instrumentation du système via **diagnostics de redondance des ouvrants**

Cette évolution permet :

- une architecture plus lisible
- une meilleure tolérance aux défaillances matérielles
- une observabilité accrue du système.
