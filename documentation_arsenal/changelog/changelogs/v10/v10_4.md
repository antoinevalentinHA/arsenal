## 📌 Ouvertures — Redondance physique N0 et réconciliation N1

v10.4 introduit une **redondance physique sur les ouvrants critiques**, absorbée
architecturalement en N1. L'API consommée par le reste du système est strictement inchangée.

### Architecture

```
N0 — Capteurs physiques (capteur_A + capteur_B)
     ↓
N1 — Réconciliation déterministe (binary_sensor.contact_*)
     ↓
N2 — Agrégation logique (binary_sensor.contact_<zone>)
```

La redondance est strictement limitée à N0. N1 reste la source canonique unique.
N2 ignore totalement la redondance physique.

### Ouvrants concernés

6 points équipés d'un capteur redondant `_bis` :

* fenêtre chambre Matthieu
* fenêtre chambre parents (gauche, milieu, droite)
* baie vitrée séjour 3
* porte garage

### Algorithme de réconciliation N1

La réconciliation est déterministe, asymétrique, fondée sur les hypothèses terrain (H1–H4) :

```
R1 — Aucune source exploitable     → unknown  (dégradé total)
R2 — Une source exploitable        → état de la source (dégradé partiel)
R3 — Deux sources, divergence      → off  (H4 : Aqara zombie dominant)
R4 — Deux sources, consensus       → état consensus
```

Un capteur `unknown` ou `unavailable` est **disqualifié** : exclu du calcul,
jamais source d'un `off`.

### Observabilité obligatoire

Chaque capteur réconcilié expose : `etat_A`, `etat_B`, `divergence`, `degrade`.
La divergence doit être visible et observable — elle ne peut être masquée.

## 📌 Diagnostic Netatmo — Refonte de la logique de qualification

* `11_template_sensors/meteo/netatmo_homekit.yaml` :

  Avant :
  * `availability` gate sur la température — capteur silencieux en cas d'indispo
  * condition réseau implicite

  Après :
  * suppression de l'`availability` gate — le capteur reste toujours évaluable
  * `ping == 'on'` est la **seule preuve réseau valide** (tout autre état → `ko_reseau`)
  * logique explicite : température numérique → `ok` / ping KO → `ko_reseau` / ping OK sans température → `ko_homekit`

  Invariant : les trois états sont structurellement stables et toujours exploitables
  par les automatisations aval, y compris en cas d'indisponibilité capteur source.

## 📌 Contrat Alarme — Acceptation contractuelle du warning scheduler

* `documentation_arsenal/contrats/alarme/20_interfaces_contexte_et_helpers.md` :

  Ajout d'un bloc normatif : `Maximum number of runs exceeded`.

  Ce warning est **accepté** dans Arsenal comme signal de backpressure du scheduler HA,
  non comme défaut métier, tant que les invariants suivants sont respectés :

  * le cerveau alarme ne consomme que des **états persistants** (states)
  * la décision est **reconstructible** à tout instant depuis les entrées contractuelles
  * les scripts d'application restent **idempotents** vis-à-vis de l'état réel

  Lignes rouges : entrée décisionnelle non persistante, dépendance à un événement unique,
  boucle de rétroaction à saturation permanente.
