## 📌 Résumé exécutif (signal net)

Cette transition est **une bascule d’architecture UI + aération**, avec en parallèle des durcissements “robustesse” sur **chauffage / ECS / volets / éclairage**.

Signal majeur :
- **UI navigation :** abandon massif de la navigation pilotée par scripts (suppression en rafale) au profit de **navigation directe** dans les dashboards + apparition de **template sensors dédiés aux cartes de navigation**.  
- **Aération / blocage chauffage :** **refonte & consolidation** (beaucoup de micro-automations supprimées → remplacement par une logique plus centralisée), et **réalignement dashboard diagnostic** vers un vocabulaire V5 (episode/blocage).  
- **Volets :** passage de commandes “groupe cover.open/close” vers **scripts unitaires** + “open = 100% garanti” (set_cover_position), avec **renommage d’entités cover** (point de risque fort).  
- **Éclairage jardin :** sortie d’un modèle “SwitchBot bouton + boolean d’état présumé” vers **commande directe de switch** (prise), + nouveaux garde-fous horaires.  
- **Suppression Browser Mod :** retrait complet du custom component → impact potentiel sur tout ce qui dépendait de popups/overlays.

(Brut : moves/renames + added/removed nets + meaningful edits.)

## 🧠 Évolutions structurantes (Most meaningful edits) — par domaine

## 📌 Chauffage / Aération / Gouvernance thermique
- **Script décision centrale chauffage :**
  - renommage “Decision centrale V2” → “Decision centrale”
  - ajout d’un **override confort** via `input_boolean.mode_confort_chauffage`
  - ajout d’un cas **aération en cours** via `input_boolean.aeration_episode_en_cours` (force reduced)
  - la “reason” est réalignée pour refléter ces nouveaux états

- **Dashboard diagnostics aération :**
  - bascule d’un retour “script” vers une **navigation directe** (navigate + path)
  - réécriture du contenu en “V5” (episode/blocage/deltaT_max…) et disparition des anciens marqueurs “reprise_thermique_*” dans la page

➡️ Lecture : tu stabilises le chauffage autour d’une **autorité plus explicite** (override confort + état aération), et tu alignes l’UI diagnostic avec cette sémantique.

---

## 📌 Volets (robustesse + scripts unitaires)
- **Scripts groupés (chambres / séjour) :**
  - remplacent `cover.open_cover`/`close_cover` multi-entity par **appel de scripts unitaires** (“100% garanti”)

- **Scripts unitaires :**
  - ouverture passe en `cover.set_cover_position` position=100 (garantie)
  - **changement d’entités** :
    - `cover.volet_arnaud` → `cover.chambre_arnaud`
    - `cover.volet_matthieu` → `cover.chambre_matthieu`
    - `cover.volet_sejour_*` → `cover.sejour_*`

➡️ Signal : durcissement “idempotence” / “position garantie”.  
➡️ Risque : **rename d’entités cover** = point de casse prioritaire.

---

## 📌 Éclairage (jardin)
- Les scripts jardin quittent le modèle “appui bouton + boolean de suivi” :
  - `off.yaml` devient **extinction conditionnelle** sur `switch.prise_jardin` (plus de boolean suivi)

- Introduction / extension des **input_datetime** (ex. heure extinction matin) pour gouverner les automatismes horaires

➡️ Signal : réduction de la dette “état présumé” → commande directe.

---

## 📌 ECS (petite maison)
- Le script “absence” devient **robuste** :
  - évite une combinaison invalide,
  - applique un mode “eco + away”,
  - vérifie localement la confirmation (state + attr) avant succès

➡️ Signal : durcissement contractuel (états invalides explicitement évités).

---

## 📌 Netatmo (fiabilisation)

➡️ Signal : découplage “stockage persistant” vs “capteur compatible stats”.
