## 🧭 Lecture système

La v12.2.2 ne modifie pas les fondations du modèle Arsenal, mais elle en
étend explicitement l’application à deux patterns désormais assumés :
la transformation contextuelle réversible des paramètres métier et
l’extraction des moteurs d’autorisation.

### Invariant 1 — Contexte = transformation réversible d'un paramètre métier

C'est le vrai apport architectural de cette version.

Jusqu'ici, le domaine Vacances agissait sur la projection de contexte global
(`input_select.mode_maison`) et sur les notifications associées. Il ne
touchait pas aux paramètres thermiques métier — par prudence, parce
qu'aucun protocole n'encadrait ce type de mutation.

La v12.2.2 introduit ce protocole. Un contexte global peut désormais
modifier un paramètre métier, à condition que la modification respecte
trois propriétés :

- **réversibilité** : la valeur précédente est sauvegardée et peut être
  restaurée sans perte ;
- **idempotence** : la transformation peut être rejouée (au démarrage, après
  une réconciliation, etc.) sans dériver — elle reconnaît son propre état
  et ne double pas ses effets ;
- **encapsulation** : la transformation est portée par un script dédié qui
  possède la transaction de bout en bout ; aucun code consommateur ne doit
  manipuler directement les paramètres métier concernés.

Dans cette version, le pattern est appliqué à la consigne réduite du
chauffage pendant les vacances. Mais il est **structurellement
indépendant** du chauffage. Il est réutilisable pour toute paire
(contexte, paramètre métier) :

- vacances × consigne VMC ;
- absence longue × cycle ECS ;
- nuit × seuils de luminosité ;
- etc.

L'implémentation concrète repose sur trois briques systématiques :

- un helper **paramètre contextuel** portant la valeur à appliquer pendant
  le contexte ;
- un helper **sauvegarde** avec sentinelle (`0`, `unknown`, ou autre
  valeur hors plage métier) servant à distinguer « aucune sauvegarde
  active » de « sauvegarde à valeur 0 » ;
- deux scripts encapsulants — un pour *entrer dans le contexte*
  (save + apply), un pour *en sortir* (restore si la sauvegarde est
  valide, no-op sinon).

C’est le pattern de référence introduit par cette version.
Il est destiné à être réemployé dans d’autres couples (contexte, paramètre métier).

### Invariant 2 — Moteur d'autorisation centralisé

Les trois automations de simulation de présence (entrée, garage, chambre
parents) portaient toutes la même condition :

```yaml
condition: or
  - input_select.mode_maison == "Vacances"
  - input_boolean.test_simulation_presence == on
```

Ce pattern viole deux principes Arsenal :

1. **Autorité dupliquée** : la logique d'autorisation est écrite trois fois,
   et n'importe quelle évolution exige trois modifications synchrones.
2. **Lecture via contexte projeté** : `input_select.mode_maison` n'est plus
   la vérité métier des vacances depuis la v12.2 — c'est sa projection.

La v12.2.2 extrait l'autorisation dans un capteur canonique :
**`binary_sensor.simulation_presence_autorisee`**. Les trois automations
lisent désormais cet unique capteur.

C'est structurellement **la même nature** que le *climatisation admissibility
lock* formalisé dans les v12.x antérieures : un capteur dédié porte la
décision « cette action est-elle autorisée en ce moment ? », et les
consommateurs se contentent de la lire. C'est aussi, dans ce cas précis,
une déduplication concrète : trois conditions `or` identiques sont remplacées
par une lecture atomique unique.

### Invariant d'exécution — Tolérance au démarrage

Au boot de Home Assistant, les entités retrouvent leurs valeurs dans un
ordre non déterministe. Pendant cette fenêtre, un capteur métier peut
produire transitoirement une valeur fausse, qui sera corrigée quelques
secondes plus tard quand ses dépendances auront elles-mêmes convergé.

Deux automations critiques de cette version — `application_debut.yaml` et
`application_fin.yaml` du domaine Vacances — formalisent un invariant
d'exécution qui s'applique plus largement :

> Toute automation qui consomme un état métier au démarrage doit absorber
> une phase de stabilisation avant d'appliquer ses effets.

Concrètement, ces automations incluent un `delay: "00:00:20"` conditionné
au trigger `homeassistant.start`. Les transitoires de boot ne déclenchent
plus l'application prématurée d'une adaptation thermique.

Cet invariant cohabite avec `input_boolean.systeme_stable` qui jouait déjà
un rôle proche pour d'autres domaines — il le complète pour les cas où la
stabilité ne peut être testée par un capteur mais doit simplement être
attendue.

### Conventions appliquées à grande échelle

En parallèle des invariants système, la v12.2.2 applique deux conventions
qui ne modifient pas le modèle Arsenal mais consolident son
implémentation :

- **Convention UI** : vingt-cinq dashboards Lovelace sont ramenés à la
  forme minimale `views: - badges: cards:`. Les métadonnées de sidebar
  (titre, icône) descendent exclusivement de `lovelace/dashboards.yaml`
  ; les `title`/`icon` locaux sont supprimés. Cette convention ne porte pas
  une règle métier, mais elle supprime un facteur de désynchronisation
  silencieuse.
- **Standard d'outillage externe** : le script Termux de restauration
  de backup (`update_from_latest_backup.sh`) est durci selon un standard
  de robustesse qui servira de grille de revue pour les futurs scripts
  externes critiques : verrou d'exécution, détection d'idempotence,
  stratégie de rollback.

Ces deux chantiers sont décrits en détail dans les sections dédiées
ci-dessous.

## 🗂️ Domaine Chauffage / Vacances

### Consignes thermiques — Troisième contexte et sauvegarde

**Nouveaux helpers :**

| Helper | Rôle | Plage |
|---|---|---|
| `input_number.chauffage_consigne_vacances` | Consigne thermique dédiée au contexte Vacances | 12–20 °C |
| `input_number.chauffage_consigne_reduite_sauvegarde` | Sauvegarde de la consigne réduite pendant la période Vacances (sentinelle `0` = aucune sauvegarde active) | 0–20 °C |

**En-tête `03_input_numbers/chauffage/consignes.yaml`** refondu : le domaine
Chauffage reconnaît désormais trois contextes de consigne (confort, réduite,
vacances) au lieu de deux.

### Scripts dédiés

**Deux nouveaux scripts** encapsulent le save/restore :

- `09_scripts/chauffage/consigne_vacances.yaml`
  → `script.vacances_adapter_consigne_reduite` : sauvegarde la réduite
  courante dans le helper de sauvegarde, puis applique la consigne vacances
  à la réduite.
- `09_scripts/chauffage/consigne_fin_vacances.yaml`
  → `script.vacances_retirer_consigne_reduite` : lit la sauvegarde ; si
  elle porte une valeur valide (différente de la sentinelle `0`), restaure
  la réduite et remet la sauvegarde à `0`.

La **sentinelle `0`** tient lieu de marqueur d'état : son rôle est de
rendre la restauration idempotente et robuste aux redémarrages. Au boot, si
`chauffage_consigne_reduite_sauvegarde == 0`, aucune restauration n'est
tentée — on ne peut pas écraser la réduite courante avec une valeur fantôme.

### Contrat formel

Nouveau contrat : `documentation_arsenal/contrats/chauffage/66_adaptation_consigne_vacances.md`.

Il formalise le protocole save/restore, la sentinelle, les cas de
réconciliation au démarrage, et les garanties d'idempotence attendues.

### Automations Vacances — Application et fin

**`10_automations/modes/vacances/application_debut.yaml` :**

- Ajout de l'appel à `script.vacances_adapter_consigne_reduite` dans le
  flux d'application.
- Ajout d'une **temporisation de 20 secondes** au démarrage (quand le
  trigger est `homeassistant.start`) — application locale de l'invariant
  *tolérance au démarrage* : les transitoires de boot ne doivent pas
  déclencher une adaptation prématurée tant que les entités métier n'ont
  pas convergé.

**`10_automations/modes/vacances/application_fin.yaml` :**

- Ajout de l'appel à `script.vacances_retirer_consigne_reduite` dans le
  flux de retrait.
- Ajout d'une **temporisation `for: 1 minute`** sur le trigger
  `binary_sensor.vacances_actives → off`. Effet : protection anti-flapping.
  Un bref passage à `off` ne retire plus instantanément l'adaptation
  thermique.
- Même `delay: "00:00:20"` conditionné au boot que `application_debut`.
- `mode: restart` → **`mode: single`** : une fois le retrait engagé, il ne
  doit pas être interrompu par une nouvelle fluctuation. C'est une décision
  de qualité d'exécution assumée — le retrait est une transaction qui se
  termine ou qui ne commence pas.
- En-tête enrichi : le rôle est explicitement *retirer les effets de
  contexte*, sans jamais fabriquer elle-même la vérité d'effectivité.

### Automation — Watchdog modification de consigne

Nouveau fichier : `10_automations/chauffage/modification_consigne.yaml`.

Cette automation introduit une couche de surveillance des changements de
consigne. Son statut contractuel reste à formaliser dans une version

## 🗂️ UI Lovelace — Convention d'implémentation

### Nature du chantier

Ce chantier n'introduit pas un invariant système — il ne porte pas de
règle métier et ne modifie pas le modèle Arsenal. C'est une **convention
d'implémentation UI** appliquée à grande échelle pour supprimer un facteur
de dette latente.

### Principe de la convention

Vingt-cinq dashboards voient leur structure ramenée à la forme minimale :

```yaml
button_card_templates: !include_dir_merge_named /config/button_card_templates

views:
  - badges: …
    cards: …
```

**Règles appliquées :**

| Élément | Avant | Après |
|---|---|---|
| `title:` au niveau racine | présent | **supprimé** |
| `icon:` au niveau racine | présent | **supprimé** |
| `title:` au niveau view | présent | **supprimé** |
| `icon:` au niveau view | présent | **supprimé** |
| `path:` au niveau view | présent (non canonique) | **supprimé** |
| Métadonnées sidebar | dupliquées | portées par `lovelace/dashboards.yaml` uniquement |

### Motivation

Le sidebar Home Assistant est alimenté exclusivement par
`lovelace/dashboards.yaml`. Les `title`/`icon` locaux des fichiers dashboards
n'étaient consommés par rien et créaient trois risques :

- désynchronisation silencieuse (un titre modifié dans un endroit, pas dans
  l'autre) ;
- illusion de vérité (on ne savait plus quel champ faisait foi) ;
- bruit de lecture.

### Fichiers concernés (liste non exhaustive)

Dashboards principaux : `arsenal.yaml`, `navigation.yaml` (implicite),
`eclairage.yaml`, `ouvertures.yaml`, `prises.yaml`, `vmc.yaml`,
`system.yaml`, `id_automatisations.yaml`.

Dashboards diagnostics : `bluetooth.yaml`, `eclairage.yaml`, `aeration.yaml`,
`alarme.yaml`, `automatisations.yaml`, `batteries.yaml`, `chauffage.yaml`,
`climatisation.yaml`, `ecs.yaml`, `mouvements.yaml`, `ouvertures.yaml`,
`thermique_chauffage.yaml`, `vannes_thermo.yaml`, `volets.yaml`,
`zigbee.yaml`.

Dashboards réglages : `chauffage.yaml`, `aeration.yaml`, `meteo.yaml`,
`eclairage.yaml`.

Dashboards imprimerie : `imprimerie.yaml`, `imprimerie_atelier.yaml`,
`imprimerie_bruit.yaml`, `imprimerie_meteo.yaml`, `imprimerie_stockage.yaml`.

Dashboards météo : `meteo_temperature.yaml`.

### Sidebar centralisé — Renommages

Dans `lovelace/dashboards.yaml`, trois titres revus pour clarté :

| Dashboard | Avant | Après |
|---|---|---|
| `arsenal-dashboard` | Arsenal | **Accueil** |
| `navigation-dashboard` | Navigation | **Menu** |

`lovelace/dashboards/reglages/eclairage.yaml` est restructuré en

- un `sub_section_header` par zone (Garage, Entrée, Séjour, Sapin) ;

Principe sous-jacent : un libellé ne duplique pas une information déjà

## 🗂️ Outillage externe — Script Termux de restauration

### Hardening complet de `update_from_latest_backup.sh`

Le script `documentation_arsenal/outils_externes/mobile/update_from_latest_backup.sh`
(Termux / Android) passe de 70 lignes basiques à 118 lignes durcies. Les
changements vont bien au-delà du polissage.

**Ajouts structurels :**

| Axe | Avant | Après |
|---|---|---|
| Gestion d'erreur | `set -e` | **`set -Eeuo pipefail`** (errtrace, undef vars, pipefail) |
| Concurrence | aucune | **verrou d'exécution** via `mkdir $LOCK_DIR` (atomique) |
| Interruption | aucun cleanup | **`trap cleanup EXIT INT TERM`** (nettoyage systématique) |
| Idempotence | aucune | **hash SHA-256 de la sauvegarde** conservé dans `$STATE_DIR/last_restored_sha256.txt` — la même sauvegarde n'est jamais restaurée deux fois |
| Rollback | `rm -rf HA/data` | **promotion `data` → `data.previous`** (ancienne arborescence conservée pour retour arrière) |
| Power management | aucun | **`termux-wake-lock`** pendant l'opération + libération au cleanup |
| Parsing de fichiers | `ls -t *.tar` | **`find -maxdepth 1 -type f -print0 | xargs -0`** (robuste aux noms de fichiers) |
| Extraction HA | `homeassistant*.tar*` | formats explicites `.tar`, `.tar.gz`, `.tgz` |
| UX | `termux-dialog` inline | helpers `error_dialog` / `info_dialog` centralisés |

### Standard d'outillage externe extrait

Ces évolutions consolident un **standard de robustesse** pour les scripts
externes critiques (hors système Arsenal lui-même) :

> Un script externe qui effectue une opération destructive (restauration,
> suppression massive, écrasement) doit disposer, au minimum : d'un verrou
> d'exécution, d'une détection d'idempotence, et d'une stratégie de
> rollback.

Ce standard n'est **pas un invariant système Arsenal** — il ne porte pas
sur le modèle métier ni sur la discipline d'écriture des entités Home
Assistant. C'est une grille de revue pour les outils externes du dossier
`documentation_arsenal/outils_externes/`. Il sert de standard pour les
futurs scripts similaires (import, export, migration, déploiement).

## ✅ Invariants — bilan

**Invariants système renforcés ou nouvellement explicités :**

- **Contexte = transformation réversible d'un paramètre métier.**
  Un contexte global peut modifier un paramètre métier à condition que la
  modification soit réversible, idempotente, encapsulée dans un script dédié.
  Pattern appliqué aux vacances × consigne réduite chauffage ; réutilisable
  pour toute paire (contexte, paramètre).
- **Un moteur d'autorisation métier est un capteur dédié, pas une
  condition dupliquée.**
  Appliqué à la simulation de présence
  (`binary_sensor.simulation_presence_autorisee`), aligné sur le pattern
  du climatisation admissibility lock.

**Invariant d'exécution renforcé :**

- **Tolérance au démarrage.**
  Toute automation consommant un état métier au boot doit absorber une
  phase de stabilisation avant d'appliquer ses effets. Appliqué aux
  automations `application_debut` et `application_fin` du domaine Vacances
  via `delay: "00:00:20"` conditionné au trigger `homeassistant.start`.

**Déjà établis, appliqués de manière étendue :**

- **L'UI lit la vérité métier, pas le contexte projeté.**
  Le dashboard diagnostic éclairage cesse d’exposer `input_select.mode_maison`
  comme pseudo-source d’autorisation et se recale sur l’autorisation canonique
  portée par `binary_sensor.simulation_presence_autorisee`.
- **Un fichier = une finalité.**
  Les scripts `consigne_vacances.yaml` / `consigne_fin_vacances.yaml` sont
  séparés, reflétant deux finalités distinctes (sauvegarde + adaptation vs
  restauration).

**Conventions et standards appliqués à grande échelle
(hors invariants système) :**

- **Convention UI** : structure minimale `views: - badges: cards:` pour les
  dashboards Lovelace. Métadonnées sidebar centralisées dans
  `lovelace/dashboards.yaml`. ~25 dashboards alignés.
- **Standard d'outillage externe** : verrou, idempotence, rollback pour
  les scripts critiques hors Arsenal. Appliqué à
  `update_from_latest_backup.sh`.

**Préservés (inchangés) :**

- **Aucune autorité décisionnelle existante n'est modifiée.**
  Le moteur thermique central et le pipeline ECS ne sont pas touchés.
- **Aucune entité métier existante n'est renommée ou supprimée dans cette
  release.** Les seuls retraits concernent des helpers de portée locale
  (`simulation_test.yaml`, `boutons.yaml`, `bots_bluetooth_stats_temp.yaml`)
  ou des fichiers obsolètes.
