## Positionnement

v12.2 marque la généralisation du modèle transactionnel Arsenal
(`request → execution → guard → verdict → trace`) à l'ensemble des
actionneurs physiques : SwitchBot, déshumidificateur, boiler.

---

## [Nouveau] Système transactionnel bots SwitchBot

Généralisation du modèle ACK chaudière aux bots physiques. Socle complet
`transactions_bots/` conçu pour être multi-domaines et extensible sans
duplication :

**Helpers** :
- Input texts : `last_action`, `last_phase`, `last_reason`, `last_request_source`,
  `last_ts`, `last_verdict`, `request_id`, `target`
- Input select : `system/transactions_bots.yaml`
- Input booleans : `lock_chambre_parents`, `lock_deshumidificateur`
- Timers : `bot_chambre_parents`, `bot_deshumidificateur`
- Compteurs : `chambre_parents`, `deshumidificateur`

**Template sensors** : `busy_*`, `cooldown_*`, `running`

**Script** : `09_scripts/system/transactions_bots.yaml`

**Contrat** : `contrats/switchbot_transactionnel.md`

---

## [Nouveau] Déshumidificateur — refonte architecture

Remplacement des automations monolithiques `on.yaml` / `off.yaml` par un
pipeline structuré en couches.

### Rupture architecturale

- suppression du modèle "script intelligent"
- interdiction de validation dans le script d'application
- introduction d'une autorité unique de vérité : le guard

**Automations** (6) : `activation`, `desactivation`, `ouverture_cycle`,
`fermeture_cycle`, `blocage_redemarrage`, `retry`

**Script `forcer_etat.yaml`** (refonte) — renommé "Appliquer état" :
script d'application pur, émet au plus une commande physique, délègue toute
qualification au guard. Ajout des champs `request_source` / `request_id`.

**Script `guard_deshumidificateur.yaml`** (nouveau) — seul juge de vérité
post-commande, retourne un verdict structuré via `input_text.*`.

**Helpers** : timers `cycle` et `blocage` ; input numbers `delai_inter_cycles`,
`duree_mini_cycle` ; input texts `debug` et `guard` ; compteur et lock.

**Contrat** : `contrats/deshumidificateur_guard.md`

## [Modification] ESPHome — BLE proxies

Activation d'un scan BLE agressif (window = interval = 1100 ms, active=true)
sur `esp32-ble-proxy-2`, `esp32-ble-proxy-3` et `esp32-ble-proxy-4`.

Objectif :
- maximiser la captation des trames BLE
- réduire les trous de couverture
- améliorer la fiabilité des bots

Effet attendu :
- latence réduite
- stabilité accrue des signaux
- augmentation de la charge radio (acceptée)
