## 🧭 Lecture stratégique

La v13.2 n'est **pas** une version de features. C'est une version de **maturation systémique**. Trois axes convergents.

### Axe 1 — Industrialisation du diagnostic système

La famille `binary_sensor.parametres_invalides_*` cesse d'être un patchwork de capteurs hétérogènes pour devenir un **socle normatif unifié**. Le pattern est désormais universel :

```
lecture pure
→ indisponible = invalide (pas d'optimisme silencieux)
→ invariants explicites et numérotés
→ attributs détaillés par source
→ cause priorisée énumérée
```

Appliqué transversalement à : éclairage, chauffage, VMC, vacances, déshumidificateur, modes maison (nouveau), bouclage ECS. C'est un **langage système** qui s'installe : « l'état du système n'est pas seulement ON/OFF, il peut être MAL CONFIGURÉ ». Cette bascule sémantique est introduite par un contrat dédié et une carte UI dédiée.

### Axe 2 — VMC : passage en logique exécutive robuste

Le sensor décisionnel `vmc_haute_vitesse_requise` est ramené à une **demande instantanée pure**. La logique de libération hystérétique migre intégralement vers l'automation `gestion_auto`, qui gagne en sécurité (trigger explicite `on/off`, garde post-délai, fallback fail-safe). Les scripts exécutifs basculent de `mode: restart` à `mode: single + max_exceeded: silent` avec délai inter-commande de 200 ms. Hardening industriel : plus d'interruption sauvage, comportement déterministe côté relais, protection électrique implicite. Un nouveau capteur `vmc_conformite_decision` et un dashboard de diagnostic VMC viennent compléter l'observabilité.

### Axe 3 — Sommeil : entrée d'un nouveau sous-système

Le domaine sommeil cesse de consommer directement les entités Withings côté pipelines aval. Un **snapshot Arsenal consolidé** (5 `input_number` + 1 `input_datetime`) devient l'unique source de vérité, alimenté par une automation de consolidation nuit. Toute la chaîne aval est réécrite (couleurs, statistics, UI) pour ne lire que ce snapshot et son drapeau de validité. Conséquence directe : les statistics passent d'un échantillonnage massif (1000–4000) à un échantillonnage cohérent avec la fréquence réelle (20/30/60). Pas un gadget : **un sous-système structuré**.

### Axe transversal — UI spécialisée

+7 dashboards nets, dont l'éclatement du dashboard météo imprimerie monolithique en quatre dashboards atomiques (température, HR, humidité absolue, humidex). Remplacement de vues fourre-tout par des vues spécialisées, cohérent avec la navigation par badges.

### Le vrai saut

> v13.1 = système propre
> v13.2 = système **auditable**

## ⚠️ Point d'attention (auto-challenge)

Le standard « indisponible = invalide » est sain mais introduit un risque latent :

- une source flaky peut devenir critique par le seul fait qu'elle est lue par un `parametres_invalides_*`
- un afflux de capteurs en `unavailable` au boot peut générer du bruit diagnostic UI

**À surveiller** : tous les helpers ne méritent peut-être pas ce niveau de sévérité. Ce point devra être réévalué si le taux de faux positifs UI au démarrage devient gênant. Piste : distinguer dans le futur les helpers « critiques » (indispo = invalide) des helpers « optionnels » (indispo = neutre).

## 2. Paramètres invalides — Pattern unifié et durcissement

**Contexte :** la famille `binary_sensor.parametres_invalides_*` était hétérogène (états calculés inline, `| float(0)` silencieux, attributs incomplets, formats de cause absents). La v13.2 standardise l'ensemble.

### Pattern commun appliqué

Chaque capteur `parametres_invalides_<domaine>` adopte désormais le même squelette :

1. **Lecture pure** des helpers via `| float(none)` ou `as_datetime(...)` après garde d'indisponibilité.
2. **Pas d'optimisme silencieux** : toute source indisponible rend l'invariant violé.
3. **Attribut `cause`** : énumération priorisée (helpers indisponibles → invariants métier dans l'ordre numéroté) avec valeur `none` si conforme.
4. **Attributs d'indisponibilité par helper** : `<helper>_indisponible` pour ≤ 5 sources, ou `helpers_indisponibles` (liste) pour > 5 sources.
5. **Bloc d'en-tête normatif** : Rôle / Principe / Invariants surveillés / Dépendances / Attributs exposés / Interdits / Entité / Statut.

### Capteurs réécrits selon ce pattern

- `11_template_sensors/system/integrite_reglages/chauffage.yaml` — invariants 1 à 5 (seuils ext OFF/ON, consignes confort/réduite, inhibition géofencing, chaîne aération M3 seuils stricte, chaîne aération M3 prolongations non stricte). 12 sources → règle `helpers_indisponibles`.
- `11_template_sensors/system/integrite_reglages/vmc.yaml` — invariant HR `seuil_on > seuil_off` ajouté à l'invariant CO₂ existant (auparavant CO₂ seul).
- `11_template_sensors/system/integrite_reglages/vacances.yaml` — invariant `fin_vacances > debut_vacances`, parsing via `as_datetime(...)` après garde, cause `fenetre_inversee`.
- `11_template_sensors/system/integrite_reglages/deshumidificateur.yaml` — invariant `cave_rh_cible_on > cave_rh_cible_off`.
- `11_template_sensors/system/integrite_reglages/eclairage.yaml` — invariants ordonnancement début < fin pour 14 helpers `input_datetime` (jardin matin + 6 cycles simulation présence).

## 3. VMC — Découplage demande / maintien et observabilité

### Sensor décisionnel — Demande instantanée pure (`11_template_sensors/vmc/haute_vitesse_requise.yaml`)

Réécriture sémantique. Le capteur cesse de porter une **logique de libération** (lecture des seuils OFF, vérification `humidite_ok and co2_ok and ha_defavorable`) et devient une **demande instantanée pure** :

```
ON  ⇔ (humidite_haute ET aeration_favorable) OU co2_haut
OFF ⇔ aucune demande instantanée
```

Suppression de la lecture des seuils OFF (`vmc_seuil_off`, `vmc_co2_seuil_off`) — déplacés dans la responsabilité de l'automation. Suppression de la lecture de l'attribut `delta_humidite_absolue` au profit de l'état direct `binary_sensor.aeration_preferable_etage`. États dégradés explicités dans l'entête (HR indispo non déclenchant, CO₂ indispo ignoré, fallback défensif HR ON 70 % / CO₂ ON 1000 ppm).

### Automation `gestion_auto` (`10_automations/vmc/gestion_auto.yaml`)

Réécriture complète. Hystérésis temporelle §6.2 portée intégralement par l'automation :

- ON → haute vitesse immédiate.
- OFF → délai `duree_min_minutes * 60` puis basse vitesse.
- Toute demande ON pendant le délai annule la descente via `mode: restart`.
- Garde supplémentaire : réévaluation de l'état OFF après délai (protège contre les cas reload / état modifié sans trigger).
- Branche `default` : retour fail-safe basse vitesse si état décisionnel indisponible.

Trigger explicité avec `to: ["on", "off"]`. Variable renommée : `duree_min` → `duree_min_minutes` (auto-documentation).

### Scripts exécutifs (`09_scripts/vmc/basse_vitesse.yaml`, `haute_vitesse.yaml`)

- `mode: restart` → `mode: single` + `max_exceeded: silent` : la séquence ne peut plus être interrompue par un nouvel appel.
- Ajout d'un `delay: 200ms` entre les deux commandes relais (anti-claquement défensif sur la transition L1/L2).

### Coherence physique (`11_template_sensors/vmc/coherence.yaml`)

Ajout de `delay_off: 2s`. Tolérance des transitoires de bascule (les écarts < 2 s sont ignorés). État ON reste valide stable, état OFF persistant uniquement.

- `10_automations/vmc/alerte_nc_decision.yaml` — **nouveau** alerteur sur non-conformité décision/état.
